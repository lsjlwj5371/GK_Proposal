# GK_Proposal — 제안서 PPT 자동 생성 파이프라인

그라운드케이(GroundK)의 제안서 PPT를 처음부터 끝까지 자동 생성하는 스킬입니다.
사용자의 클라이언트 분석 자료를 받아 구조화하고, AI로 브랜드 색상 추출 → 카피 정제 → pptxgenjs 코드 생성 → 시각 QA까지 수행합니다.

## 핵심 원칙

1. **콘텐츠 우선 원칙**: 템플릿보다 논리 구조를 먼저 설계한다
2. **가이드 기반 정밀 매칭**: 표지/간지/마무리는 가이드에 정의된 실측 좌표를 정확히 따른다
3. **대화형 진행**: 정보가 부족하면 사용자에게 물어보고, 단계별 확인을 받으며 진행한다
4. **4분류 체계**: 모든 페이지는 표지/간지/본문-기본/본문-강조/마무리로 분류한다
5. **생성 후 자체 검수 필수**: PPT 생성 후 반드시 코드 레벨에서 전 슬라이드를 검수하고, 문제를 수정한 뒤에 최종 결과물을 사용자에게 전달한다

| **제작시나리오 가이드** | `references/02_제작시나리오_가이드.md` | 43개 템플릿 설명, 조합 패턴, 품질 체크리스트 | Phase 2 템플릿 매핑 시 |

## 코드 패턴 라이브러리 (검증된 레이아웃 컴포넌트)

실제 우수 PPT 레퍼런스 18장을 분석하여 추출한 **파라미터화된 적응형 컴포넌트**입니다. 고정 템플릿이 아니라, 콘텐츠 양/수에 따라 자동으로 그리드·폰트·간격이 조절됩니다.

### 컴포넌트 목록

| 컴포넌트 | 경로 | 변형 | 용도 |
|---------|------|------|------|
| **표지** | `patterns/components/cover.js` | A(미니멀), B(시네마틱) | 표지 페이지 |
| **간지** | `patterns/components/divider.js` | A(센터), B(사이드바), C(이미지분할), D(브랜드그래픽) | 챕터 구분 |
| **카드 그리드** | `patterns/components/card_grid.js` | default, dark, kpi, image-overlay | 본문 카드 레이아웃 |
| **강조 본문** | `patterns/components/highlight_body.js` | dark-keywords, image-overlay-message, split-impact | 클라이맥스 페이지 |
| **마무리** | `patterns/components/ending.js` | A(미니멀), B(시네마틱), C(브랜드컬러) | 감사/다짐 페이지 |

### 패턴 선택 흐름

```
Phase 2에서 콘텐츠 구조화 완료
    ↓
pattern_index.json의 moodMapping에서 제안서 분위기에 맞는 패턴 세트 선택
    ↓
각 페이지의 콘텐츠 유형에 따라 contentRecommendations 참조
    ↓
선택된 컴포넌트를 require()로 로드하여 사용
    ↓
컴포넌트가 카드 수, 텍스트 양에 따라 자동 적응
```

### 사용 예시

```javascript
const cardGrid = require('./patterns/components/card_grid');
const createDivider = require('./patterns/components/divider');
const createCover = require('./patterns/components/cover');

// 표지 (미니멀 타입)
createCover(sl, pptx, {
  type: 'A', title: '제안서 제목', companyName: '(주)그라운드케이',
  palette, label: '정성적 제안서'
});

// 간지 (사이드바 타입 + 하위 목차)
createDivider(sl, pptx, {
  type: 'B', chapterNum: '01', chapterTitle: '제안 개요',
  subItems: ['사업 배경', '제안 범위', '추진 방향'], palette
});

// 카드 그리드 (KPI 스타일 — 큰 숫자 강조)
cardGrid(sl, pptx, {
  style: 'kpi', palette,
  cards: [
    { number: '35%', title: 'GRP 수익 전환', body: '상금 체계 → 글로벌 수익 풀 도입' },
    { number: '2M+', title: '팬 도달 규모', body: '디지털 플랫폼 기반 글로벌 팬덤' },
  ]
});
```

> **핵심 원칙**: 패턴의 "뼈대"(그리드, 비율)는 콘텐츠에 따라 유동적으로 변하고, "피부"(그림자, 둥근 모서리, 색상, 오버레이)만 레퍼런스에서 검증된 것을 고정 적용한다.

## 아이콘 시스템 (Lineicons SVG)

606개 Lineicons SVG 아이콘이 `assets/icons/svgs/`에 포함되어 있습니다. 이모지 대신 SVG 아이콘을 사용하여 프로페셔널한 품질을 확보합니다.

| 파일 | 경로 | 역할 |
|------|------|------|
| **SVG 아이콘** | `assets/icons/svgs/*.svg` | 606개 Lineicons (MIT 라이선스) |
| **아이콘 매니페스트** | `assets/icons/icon_manifest.json` | 한/영 키워드 매핑, 카테고리, 용도 설명 |
| **아이콘 헬퍼** | `scripts/icon_helper.js` | SVG 로드, 색상 변환, PNG 변환 유틸리티 |

### 아이콘 사용 흐름

```javascript
const { findIcons, iconToBase64PNG } = require('./scripts/icon_helper');

// 1. 콘텐츠 키워드로 아이콘 검색
const matches = findIcons('전략', { limit: 3 });
// → [{ file: 'rocket-5.svg', keywords: [...], ... }]

// 2. 색상을 브랜드 팔레트에 맞춰 PNG 변환
const iconData = await iconToBase64PNG('rocket-5.svg', {
  color: '#FFFFFF',  // 다크 배경용 흰색
  size: 64
});

// 3. PPT에 이미지로 삽입
sl.addImage({ data: iconData, x: 1.0, y: 2.0, w: 0.5, h: 0.5 });
```

> **아이콘 색상 변환 원리**: 원본 SVG의 `fill="#323544"`를 원하는 색상으로 교체한 뒤 PNG로 변환. 웹사이트에서 색상 선택하는 것과 동일한 결과를 코드로 구현.

## 전체 파이프라인 (5 Phase)

```
사용자 요청 → Phase 0: 자료 수집/인터뷰
                ↓
           Phase 1: 브랜드 분석 (색상 + 배경)
                ↓
           Phase 2: 콘텐츠 구조화 (논리설계 → 템플릿 매핑)
                ↓
           Phase 3: PPT 코드 생성 (pptxgenjs)
                ↓
           Phase 3.5: 자체 검수 (겹침/정렬/색상대비/분류차별화/폰트) ← 필수
                ↓     문제 발견 시 수정 후 재생성
           Phase 4: 시각 QA (필수 — PowerPoint COM + Claude 직접 vision 분석)
                ↓
           최종 .pptx 파일 전달
```

## Phase 0: 자료 수집 및 인터뷰

사용자의 요청을 받으면, 아래 정보를 파악합니다. 이미 제공된 정보는 건너뛰고, 부족한 부분만 질문합니다.

### 필수 수집 항목

```
1. 기본 정보: 사업명, 발주처, 사업기간, 예산규모
2. 배경/현황: 현재 과제, 시장 환경
3. 제안 내용: 솔루션, 핵심 기능, 차별점
4. 실적: 유사 프로젝트, KPI
5. 조직: 투입 인력, 구성
6. 비용/일정: 견적, 마일스톤
7. 회사 정보: 연혁, 비전
8. 제안 대상 브랜드: 회사명, 도메인(로고 추출용), CI 색상
```

### 자료 부족 시 처리 (A/B/C 등급)

수집한 정보의 충분도를 평가합니다:

| 등급 | 기준 | 대응 |
|------|------|------|
| **A** | 핵심 콘텐츠 충분 | 바로 생성 진행 |
| **B** | 골격은 있으나 세부 부족 | 공개 정보로 보강 가능한 부분 보강 후 사용자 확인 |
| **C** | 핵심 정보 누락 | 사용자에게 추가 자료 요청 (구체적으로 무엇이 필요한지 안내) |

### 인터뷰 방식

- 한 번에 너무 많은 질문을 하지 않는다 (최대 3개씩)
- 이미 제공된 파일(PDF, DOCX 등)이 있으면 먼저 읽고 분석한 뒤, 부족한 부분만 질문
- "~에 대한 자료가 있으신가요?" 식으로 부드럽게 질문
- 사용자가 "알아서 해줘"라고 하면 공개 정보 기반으로 최선의 판단 후 확인 요청

## Phase 1: 브랜드 분석

`scripts/ai_brand_analyzer.js`를 활용하여:

1. **로고 추출**: 제안 대상 도메인에서 로고 이미지 가져오기
2. **색상 추출**: 로고에서 dominant color 추출
3. **6:3:1 팔레트 생성 (Adobe Color Harmony 기반)**:
   - DOMINANT (60%): 브랜드 CI 색상 그대로 사용 — 배경, 간지, 넓은 면적
   - SECONDARY (30%): **DOM과 독립적인 조화색** — 카드, 구분선, 태그
   - ACCENT (10%): **DOM과 독립적인 강조색** — 핵심 수치, CTA
   - 파생: DARK_BG, LIGHT_BG, CARD_BG, SUBTLE (DOM 기반으로 통일감 유지)
4. **배경 이미지 생성**: 표지/간지/마무리용 그라디언트+장식 배경 (SVG→PNG)

### ★ 색상 조화 규칙 (매우 중요)

SEC/ACC는 브랜드 색상의 밝기·채도 변형이 **아닌**, 색상환(Color Wheel)에서 조화 규칙에 따라 선택된 **독립적인 색상**이다.

| 규칙 | 색상환 각도 | 느낌 | 적합한 제안서 |
|------|-----------|------|-------------|
| **Split-Complementary** (기본값) | SEC: +150°, ACC: +210° | 세련된 대비 | 일반 비즈니스 |
| **Triadic** | SEC: +120°, ACC: +240° | 역동적, 활기찬 | 스포츠, 이벤트 |
| **Complementary** | SEC: +180°, ACC: +150° | 강한 대비, 주목도 | KPI 강조, 성과 |
| **Analogous + Complement** | SEC: +30°, ACC: +180° | 안정감 + 포인트 | 공식 제안서 |
| **Square** | SEC: +90°, ACC: +270° | 다채롭고 풍부한 | 문화 행사, 크리에이티브 |

Phase 0에서 제안서의 주제/분위기를 파악한 뒤, `mood` 파라미터로 적절한 조화 규칙을 자동 선택한다. 사용자가 수동으로 지정할 수도 있다.

### 색상 확인

팔레트 생성 후 사용자에게 결과를 보여주고 확인을 받는다:
```
제안 대상: [회사명]
DOMINANT: #XXXXXX (설명)
SECONDARY: #XXXXXX (설명)
ACCENT: #XXXXXX (설명)

이 색상 조합으로 진행할까요? 수정이 필요하면 말씀해주세요.
```

## Phase 2: 콘텐츠 구조화

### ★ 최우선 원칙: 콘텐츠 깊이가 디자인 밀도를 결정한다

페이지가 비어 보이는 가장 큰 원인은 디자인이 아니라 **콘텐츠 부족**이다.
아이콘, 사진, 장식 도형으로 빈 공간을 채우는 것은 보조 수단일 뿐이며,
**충분하고 구체적인 내용**이 페이지를 채우는 것이 가장 좋은 디자인이다.

**따라서 Phase 3(PPT 코드 생성)에 들어가기 전에, 모든 페이지의 콘텐츠가 아래 기준을 충족하는지 반드시 확인한다.**

### Step 0: 사용자 자료 심층 분석 (Phase 2의 전제)

사용자가 제공한 분석 자료(DOCX, PDF, 텍스트 등)를 **최대한 깊이 있게** 읽고 핵심을 추출한다.

**자료 분석 체크리스트:**
1. 자료에 포함된 **모든 수치/데이터**를 추출 (시장 규모, 성장률, 비율, 금액, 인원 등)
2. 자료에 언급된 **핵심 키워드/개념**을 목록화
3. 자료의 **논리 구조**(문제→원인→해결→기대효과)를 파악
4. 자료에 **명시적으로 나오지 않지만 추론 가능한 인사이트**를 도출
5. 자료의 근거를 바탕으로 **구체적 실행 방안/전략**을 구체화

**자료가 부족할 때 콘텐츠 보강 방법 (우선순위순):**
1. **사용자 자료에서 더 깊이 추출**: 한 문장에서도 카드 2~3장 분량의 콘텐츠를 도출 가능
   - 예: "GRP 도입" → ① GRP의 정의 ② 기존 상금 체계와의 차이 ③ 도입 시 기대 수익 구조 변화
2. **공개 정보 보강**: 웹 검색으로 관련 시장 데이터, 사례, 통계를 보완
3. **논리적 확장**: 자료의 방향성을 근거로 세부 전략, 실행 계획, 기대효과를 구체화
4. **비교/대조 추가**: 업계 벤치마크, 경쟁사 사례, Before/After 비교
5. **마지막 수단**: 위의 모든 방법으로도 콘텐츠가 부족하면 사용자에게 추가 자료 요청

### Step 1: 논리 구조 설계 (콘텐츠 우선)

템플릿을 먼저 고르지 않는다. 제안서의 스토리라인을 먼저 설계한다:

1. 전체 목차 구성 (챕터 나누기)
2. 각 페이지의 역할과 핵심 메시지 정의
3. 논리적 흐름 검증 (WHY → WHAT → HOW → PROOF → PROMISE)

### Step 1.5: 페이지별 콘텐츠 깊이 검증 (Step 1과 Step 2 사이 — 필수)

각 페이지에 배치할 콘텐츠의 양과 구체성을 점검한다. **이 검증을 통과하지 못하면 Step 2로 넘어가지 않는다.**

**페이지별 최소 콘텐츠 기준:**

| 페이지 유형 | 최소 콘텐츠 |
|-----------|-----------|
| **카드 나열형** (2~4개 카드) | 각 카드에 제목(1줄) + 본문(2~4줄) + 수치 또는 근거 1개 이상 |
| **프로세스/타임라인** | 각 단계에 명칭 + 설명(1~2줄) + 기대효과 또는 산출물 |
| **KPI/수치 강조** | 수치 + 수치의 의미 설명(1줄) + 비교 기준(전년 대비, 업계 평균 등) |
| **전략/방향** | 전략 명칭 + 왜 이 전략인지(배경 1~2줄) + 구체적 실행 방안(2~3줄) |
| **비교/분석** | 비교 항목 3개 이상 × 비교 대상 2개 이상 |
| **사례/레퍼런스** | 사례 명칭 + 핵심 내용(2줄) + 시사점(1줄) |

**콘텐츠가 기준에 미달하면:**
- 해당 페이지의 콘텐츠를 보강한다 (Step 0의 보강 방법 활용)
- 보강이 불가능하면 해당 페이지를 다른 페이지와 합치거나 삭제한다
- **절대 콘텐츠가 부족한 상태로 디자인 단계에 넘기지 않는다**

**콘텐츠 깊이 예시 — 같은 주제, 다른 깊이:**

```
[나쁜 예 — 얕은 콘텐츠]
카드 제목: "LCK 수익 구조 전환"
카드 본문: "상금 체계 폐지 → GRP 도입"
→ 카드 내부가 비어 보임, 디자인으로 해결 불가

[좋은 예 — 깊은 콘텐츠]
카드 제목: "LCK 수익 구조 전환"
카드 수치: "상금 → GRP 전환"
카드 본문:
  • 기존: 대회 상금 중심 (연 50억 규모)
  • 변화: GRP(글로벌 수익 풀) 기반 배분
  • 효과: 구단 수익 안정화, 대기업 스폰서십 고도화
  • 선례: 한진·키움증권·우리은행 참여 확정
→ 카드가 자연스럽게 꽉 참, 디자인 보조 최소화
```

### ★ 본문-강조 페이지 디자인 원칙 (매우 중요)

강조 페이지는 제안서의 **클라이맥스**에 해당하는 페이지다. 다른 본문 페이지들과 시각적으로 확연히 구별되어야 한다.

**강조 페이지가 배치되어야 하는 콘텐츠:**
- "우리가 이것을 하겠다" — 핵심 제안/솔루션 선언
- "이 프로젝트는 이런 방향이다" — 비전/방향성 제시
- 핵심 KPI/성과 수치 — 임팩트 있는 숫자로 설득하는 장면
- 전략의 핵심 전환점 — 스토리라인에서 가장 중요한 메시지

**강조 페이지에 부적합한 콘텐츠:**
- 레퍼런스/포트폴리오 갤러리 (이것은 본문-기본)
- 단순 데이터 테이블, 일정표
- 조직도, 팀 소개

**시각적 차별화 방법 (반드시 하나 이상 적용):**
1. **배경 색상 반전**: 흰색 배경 대신 DOMINANT 또는 DARK_BG를 전면 배경으로 사용, 텍스트를 밝은 색으로 반전
2. **대형 다크/컬러 블록**: 콘텐츠 영역 전체를 채우는 대형 배경 블록 + 밝은 텍스트
3. **배경 이미지 + 오버레이**: 관련 이미지 위에 반투명 DOM 오버레이를 깔고 흰색 텍스트
4. **대형 타이포그래피**: 핵심 수치나 메시지를 48pt+ 초대형으로 배치

강조 페이지라도 **헤더 영역(페이지번호, 핵심메시지, 부제목, 구분선)은 동일 좌표로 유지**한다. 배경이 어두울 경우 헤더 텍스트 색상만 밝게 반전한다.

### ★ 간지(Divider) 페이지 — 표지/마무리와의 차별화 원칙

간지는 챕터 전환을 알리는 페이지로, 표지/마무리와 배경색·디자인이 달라야 시각적 구분이 된다.

| 구분 | 배경 | 톤 |
|------|------|-----|
| **표지/마무리** | DARK_BG 또는 DOMINANT (가장 어둡고 무거운 톤) | 선언적, 임팩트 |
| **간지** | SECONDARY 또는 DOMINANT의 밝은 변형 (표지보다 밝거나 색상이 다른 톤) | 전환, 호흡 |

간지가 표지/마무리와 동일한 배경색을 사용하면 제안서의 리듬감이 사라진다. 반드시 시각적으로 구분되는 색상·패턴을 적용할 것.

#### ★ 간지 다양성 강제 — 한 제안서 내 간지 variant 중복 금지

한 제안서에 간지가 **2개 이상** 등장하면 서로 다른 type을 사용해야 한다. 4개 챕터 모두 같은 type(예: 사이드바)을 쓰면 시각적 단조로움이 극심해진다.

| 챕터 수 | 권장 type 조합 |
|--------|--------------|
| 2 | A(센터) + B(사이드바) |
| 3 | A(센터) + B(사이드바) + C(이미지분할) |
| 4 | A + B + C + D(브랜드그래픽) |
| 5+ | 위 4종 순환, 단 연속 동일 type 금지 |

**규칙:**
- 같은 type을 두 번 사용하지 않는다 (콘텐츠 5+ 챕터 외)
- **연속된 두 간지는 절대 동일 type 사용 금지**
- 간지의 배경 색상도 가급적 톤 변화를 줘서(DOM ↔ SEC ↔ DOM의 밝은 변형) 리듬을 만든다

#### ★ 강조 페이지 다양성 강제 — 한 제안서 내 highlight variant 중복 금지

마찬가지로 본문-강조 페이지가 2개 이상이면 서로 다른 variant를 사용한다.

| 강조 페이지 수 | 권장 variant 조합 |
|------------|----------------|
| 2 | dark-keywords + image-overlay-message |
| 3 | dark-keywords + image-overlay-message + split-impact |
| 4+ | 위 3종 순환, 연속 동일 variant 금지 |

같은 variant를 반복하면 "두 번째 강조"가 첫 번째와 동일해 보여 클라이맥스의 의미가 사라진다.

### ★ 작업 디렉토리 정책 (필수)

PPT 생성 작업 폴더는 **반드시 아래 경로 하위에** 시나리오별 서브폴더로 생성한다. Temp 폴더(`%TEMP%`, `AppData\Local\Temp`)는 사용 금지.

```
베이스 경로:
C:\Users\akstn\OneDrive\바탕화면 - DESKTOP-G1P5I6M (334012)\이우재\01. AX Project\02. 제안서 자동화\클로드 작업파일\

시나리오별 폴더:
└── {YYYYMMDD}_{시나리오슬러그}_v{N}\

    ├── slides_png/           ← Phase 4 시각 QA용 PNG
    ├── {제안서명}.pptx       ← 최종 산출물
    └── (기타 작업 파일들)
```

**규칙:**
1. **베이스 경로 변수**로 코드 상단에 정의:
   ```javascript
   const WORK_BASE = 'C:\\Users\\akstn\\OneDrive\\바탕화면 - DESKTOP-G1P5I6M (334012)\\이우재\\01. AX Project\\02. 제안서 자동화\\클로드 작업파일';
   ```
2. **폴더명 형식**: `YYYYMMDD_시나리오슬러그_v버전` (예: `20260410_samsung_pyeongtaek_v1`)
   - 시나리오슬러그는 영문 소문자 + 언더스코어
   - 같은 날 같은 시나리오를 재생성하면 v2, v3로 증가
3. **폴더 미존재 시 생성**: `fs.mkdirSync(workDir, { recursive: true })`
4. **경로 처리 주의**:
   - Bash 환경에서 OneDrive 한글 경로는 종종 `바탕화~1-DESKTOP-G1P5I6M-334012` 같은 단축 경로로 표시되지만, **Node.js / PowerShell에서는 원본 한글 경로를 사용**한다
   - 경로에 공백이 있으므로 PowerShell 호출 시 반드시 따옴표 처리
5. **사용자에게 안내**: 작업 시작 시 "작업 폴더: `클로드 작업파일\YYYYMMDD_xxx_v1`" 한 줄로 통지

const FN_XB = 'Pretendard ExtraBold';

**마무리 (createEnding) — S43:**
```
LEFT-ALIGNED x=1.47"
         ★ 히어로 = 다짐/약속 카피 (36pt ExtraBold, y=3.40")
           → "감사합니다"가 아니라 "성공적인 대회를 약속드립니다" 같은 약속 문구가 주인공
         구분선 y=5.30" w=2"
         "감사합니다" y=5.63" 14pt Medium (보조 텍스트로 격하)
         저작권 y=6.45" 12pt Thin
         bgData 파라미터로 AI 배경 적용
```

> **마무리 페이지 히어로 규칙**: 마무리 슬라이드의 가장 큰 텍스트(히어로)는 **약속/다짐 카피**여야 한다. "감사합니다"는 보조 텍스트(14pt)로 아래에 작게 배치한다. 우수 레퍼런스(BIFF, APEC 등)를 분석한 결과, 모든 마무리 페이지에서 히어로는 약속 메시지이고 "감사합니다"는 서포팅 역할이었다.

> **마무리 히어로 텍스트박스 자동 검증 (필수)**: 히어로 텍스트박스가 좁아서 글자가 세로로 1자씩 배치되거나 슬라이드 경계를 벗어나는 사고를 방지한다.
>
> **계산 공식**:
> - 한글 1자 점유 폭(인치) ≈ `폰트사이즈(pt) × 0.6 ÷ 72`
> - 필요 width ≈ `글자수 × 한글 1자 폭 + 패딩 0.5″`
> - 예: 36pt × 12자 = `36 × 0.6 ÷ 72 × 12 + 0.5 ≈ 4.1″`
>
> **검증 규칙**:
> 1. 히어로 텍스트박스 width는 슬라이드 width의 **60% 이상(최소 7″)** 확보 — 좁게 1~2″로 배치 금지
> 2. 위 공식으로 계산한 필요 width가 textbox width를 초과하면 → 폰트 자동 축소 또는 줄바꿈 강제
> 3. 텍스트박스 height는 폰트사이즈 × 1.5 × 줄수 ÷ 72 + 0.3″ 이상

**본문 헤더 (모든 본문 페이지 공통 — 절대 고정):**
```
챕터라벨: x=0.47" y=0.21" w=5.0" h=0.25"  (9pt Medium)
  → 형식: "Ch1. 산업 이해와 과제 진단" (챕터번호 + 챕터명으로 통일)
  → ❌ "03  산업 이해" (페이지 번호 사용 금지)
핵심 메시지 (★ExtraBold 16pt): x=0.47" y=0.68" w=10.75" h=0.47"  ← 반드시 FN_XB 적용
부제목 (Regular 10pt): x=0.47" y=1.18" w=10.75" h=0.29"
구분선 (ACCENT): x=0.47" y=1.55" w=10.75" h=0.015"
콘텐츠 시작선: y=1.77" (Content Box 상단)
콘텐츠 영역: Content Box 참조 — x=0.53~11.16" (w=10.63"), y=1.77~7.68" (h=5.91")
```

> **헤드라인 폰트 규칙**: 각 본문 페이지의 핵심 메시지(y=0.68 위치)는 반드시 `Pretendard ExtraBold` (FN_XB)를 사용한다. SemiBold나 Bold가 아닌 ExtraBold여야 제안서의 톤이 유지된다.

> **챕터 라벨 규칙**: 좌측 상단 라벨은 해당 페이지가 속한 챕터의 번호와 챕터명을 표시한다. 같은 챕터에 속한 모든 페이지는 동일한 라벨을 사용한다. 페이지 순번(03, 04...)이 아닌 챕터 정보를 보여주는 것이 목적이다.

**슬라이드 규격**: A4 가로 11.69 × 8.27 inches

### ★ 콘텐츠 박스 (Content Box) — 본문 페이지 레이아웃의 핵심 원칙

모든 본문 페이지(기본+강조)에서, 헤더 아래의 콘텐츠는 **가상의 사각형 영역(Content Box)** 안에 균형감 있게 배치한다. 이 박스는 실제로 그려지는 도형이 아니라, 콘텐츠 배치의 기준이 되는 논리적 영역이다.

```
┌─────────────────── 슬라이드 (11.69" × 8.27") ──────────────────┐
│  ┌──── 헤더 (y=0.21~1.55) ────┐                                │
│  │ 페이지번호 · 핵심메시지 · 부제목 · 구분선                      │
│  └─────────────────────────────┘                                │
│                                                                  │
│  ┌════════════ Content Box (10.63" × 5.91") ════════════┐       │
│  ║  x=0.53"                                    x=11.16" ║       │
│  ║  y=1.77"                                              ║       │
│  ║                                                       ║       │
│  ║   모든 콘텐츠(텍스트, 카드, 차트, 아이콘, 도표)를       ║       │
│  ║   이 영역 안에 균형감 있게 배치한다                     ║       │
│  ║                                                       ║       │
│  ║                                                       ║       │
│  ║  y=7.68"                                              ║       │
│  └═══════════════════════════════════════════════════════┘       │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

**Content Box 규격:**
| 항목 | 값 | 비고 |
|------|-----|------|
| 너비 | 10.63" (27cm) | 슬라이드 좌우 중앙 정렬 |
| 높이 | 5.91" (15cm) | 헤더 구분선 아래부터 |
| X 시작 | 0.53" | (11.69 - 10.63) / 2 |
| Y 시작 | 1.77" | 헤더 구분선(y=1.55) + 여백 0.22" |
| X 끝 | 11.16" | 0.53 + 10.63 |
| Y 끝 | 7.68" | 1.77 + 5.91 |

**Content Box 적용 규칙:**
1. **경계 준수**: 모든 텍스트, 카드, 아이콘, 차트는 Content Box 경계를 벗어나지 않는다
2. **균등 배치**: 카드나 아이콘이 여러 개일 때, Content Box 내에서 균등 간격으로 배치한다
3. **여백 일관성**: Content Box 내부의 상하좌우 패딩은 최소 0.15"를 유지한다
4. **슬라이드 간 통일성**: 서로 다른 본문 페이지라도 콘텐츠의 시작점(x, y)과 끝점이 Content Box 기준으로 정렬되어 일관된 리듬감을 준다
5. **그리드 활용**: Content Box를 2등분(좌/우), 3등분, 4등분 등으로 나눠 레이아웃 그리드로 활용한다

**레이아웃 그리드 예시:**
```javascript
// Content Box 기본 상수
const CB_X = 0.53;   // Content Box X 시작
const CB_Y = 1.77;   // Content Box Y 시작
const CB_W = 10.63;  // Content Box 너비
const CB_H = 5.91;   // Content Box 높이
const CB_PAD = 0.15; // Content Box 내부 패딩
const CB_GAP = 0.20; // 카드 간 간격

// 2열 그리드
const col2W = (CB_W - CB_PAD * 2 - CB_GAP) / 2;
const col2_x1 = CB_X + CB_PAD;
const col2_x2 = col2_x1 + col2W + CB_GAP;

// 3열 그리드
const col3W = (CB_W - CB_PAD * 2 - CB_GAP * 2) / 3;
const col3_x1 = CB_X + CB_PAD;
const col3_x2 = col3_x1 + col3W + CB_GAP;
const col3_x3 = col3_x2 + col3W + CB_GAP;
```

> **주의**: 표지(S01), 간지(S02), 마무리(S43)에는 Content Box를 적용하지 않는다. 이들은 가이드에 정의된 고정 좌표를 따른다.

### ★ 안티패턴 — 절대 하지 말 것 (매우 중요)

**[AP-01] 밝은 배경 위 어두운 대면적 요소 금지 — 정량 기준 25%**

본문 페이지(흰색 배경)에서 카드·블록을 어두운 색(DOM, DK 등)으로 채우면 시각적 무게감이 과도해진다. 핵심은 **대비의 방향**: 어두운 배경+밝은 카드는 OK, 밝은 배경+어두운 카드는 NG.

> **★ 정량 기준**: 본문 페이지(밝은 배경) 콘텐츠 영역(Content Box, 62.82 in²) 내에서 **단일 다크 블록(DOM/DK 색상)의 면적이 25%(약 15.7 in²)를 초과하면 위반**으로 간주한다. 25% 미만이라도 좌우/상하 균형을 시각적으로 해치는 비대칭 배치는 그레이존으로 판정.

| 조합 | 판정 | 이유 |
|------|------|------|
| 어두운 배경 + 밝은 카드 | ✅ 좋음 | 다크가 프레임 역할, 밝은 카드가 떠보이는 느낌 |
| 밝은 배경 + 어두운 카드(25% 초과) | ❌ 위반 | 흰 페이지 위에 어두운 덩어리가 무겁게 내려앉음 |
| 밝은 배경 + 어두운 카드(25% 이하, 균형 OK) | 🟡 그레이존 | 좌우 비대칭/상하 무게감 검토 필수 |
| 밝은 배경 + 밝은 카드 | ✅ 기본 | 그림자/보더로 카드 구분. 가장 안정적 |

**위반 시 대응**:
1. 콘텐츠 영역 전체 배경을 다크로 전환 → "어두운 배경 + 밝은 카드" 구조로 재구성
2. 다크 블록을 ACC 컬러 라인 + 밝은 카드 + 그림자로 대체
3. 다크 블록 면적을 25% 이하로 축소하고 좌우 균형 재조정

```
❌ 나쁜 예: card-grid의 'dark' 스타일을 흰 배경 본문에서 사용
  → 콘텐츠 영역 전체가 남색/검정으로 가득 차 밸런스 붕괴
  → 이런 경우 카드에 이미지+그라데이션 오버레이를 적용하거나,
    배경 자체를 어둡게 바꿔서 "어두운 배경 + 밝은 카드" 구조로 전환

✅ 좋은 대안:
  1. 밝은 카드(CD, LT, WHITE) + 그림자 + 액센트 바로 구분
  2. 카드에 이미지 배경 + 좌→우 그라데이션 오버레이
  3. 배너(상단 40% 다크) + 하단 밝은 카드 (bannerCards 패턴)
```

**[AP-02] 하드 컬러 분할(hard split) 금지**

슬라이드 상반부 흰색 + 하반부 검은색 → 페이지가 둘로 잘려 보이는 느낌. split-impact 등에서 하단 다크 블록을 사용할 때 반드시 완화 처리한다.

```
❌ 나쁜 예: 상단 50% 흰색 + 하단 50% 순수 검정 → 페이지가 잘림
✅ 좋은 대안:
  1. 하단 블록에 transparency: 10~20 적용하여 부드럽게
  2. 분할 비율을 70:30으로 (다크 영역 축소)
  3. 분할선에 그라데이션 전환 또는 ACC 색상 라인
  4. 하단에 이미지 배경 + 반투명 오버레이 사용
```

**[AP-03] 빈 장식 도형 금지**

원형(OVAL), 사각형 등 장식 도형을 배치할 때, 내부가 비어있으면 **미완성 슬라이드**로 보인다.

```
❌ 나쁜 예: 원형 도형만 덩그러니 놓여 있고, 내부에 아이콘/텍스트 없음
✅ 반드시 내부에 다음 중 하나를 포함:
  - SVG 아이콘 (icon_helper.js로 변환)
  - 번호/숫자 텍스트
  - 의미 있는 라벨 텍스트
  - 이미지
  → icon이 없으면 label의 첫 글자를 크게 넣는 등 폴백 처리 필수
```

**[AP-04] 배너/히어로 영역 텍스트만 배치 금지**

배너(banner) 또는 히어로 영역에 텍스트만 넣으면 상단이 비어 보인다.

```
❌ 나쁜 예: 다크 배너 영역에 숫자+설명 텍스트만 존재
✅ 반드시 아이콘, 장식 도형, 이미지 플레이스홀더 중 하나 이상 함께 배치
```

**[AP-05] 텍스트박스 오버플로우 / 강제 줄바꿈 금지**

텍스트박스의 width/height가 콘텐츠 길이를 수용하지 못해 텍스트가 도형 밖으로 빠져나가거나, 의도치 않은 위치에서 줄바꿈이 발생하는 경우.

| 증상 | 예시 |
|------|------|
| 라벨 도형 안 텍스트가 2줄로 줄바꿈 | "임직원 통근 셔틀 제안서" 가 표지 라벨 도형에서 한 줄 → 두 줄 |
| 챕터 라벨이 슬라이드 좌측 경계 밖으로 빠져나감 | "CHAPTER" 텍스트박스 x좌표 음수 |
| 카드 본문이 카드 box 밖으로 흘러나감 | 텍스트박스 height 부족 |
| 히어로 타이포가 텍스트박스 width 부족으로 세로 1자씩 배치 | 마무리 페이지에서 빈번 |

**필수 조치**:
1. **라벨/히어로 도형은 width를 텍스트 길이에 맞춰 가변 처리**: 도형 width = 글자수 × 폰트사이즈(pt) ÷ 72 × 0.6 (한글 기준 보정) + 패딩 0.3″
2. **autoFit/shrinkText 적용**: pptxgenjs `addText`의 `fit: 'shrink'` 또는 `autoFit: true` 옵션 사용
3. **마무리 히어로**: 텍스트박스 width를 슬라이드 width의 60% 이상으로 확보 (최소 7″) — 절대 1~2″로 좁게 배치 금지
4. **검증**: 자체 검수에서 글자수 × 폰트사이즈로 텍스트 점유 폭을 계산해 textbox width를 초과하지 않는지 확인

**[AP-06] 카드 그룹 내 폰트 크기 불일치 금지**

같은 그리드(2x2, 3x1 등)에 속한 카드들은 콘텐츠 길이가 달라도 헤드라인/본문 폰트 크기가 **모두 동일**해야 한다.

```
❌ 나쁜 예: 카드 1=14pt, 카드 2=12pt, 카드 3=11pt (콘텐츠 길이에 따라 임의 조정)
✅ 원칙: 그리드 내 가장 긴 카드의 콘텐츠 기준으로 폰트를 결정하고, 모든 카드에 동일하게 적용
   짧은 카드는 폰트가 같은 채로 빈 공간이 생기면 → 콘텐츠 보강 또는 도식화로 채움
```

**[AP-07] 카드 그룹 내 좌측 정렬 불일치 금지**

그리드 내 카드의 헤드라인, 구분선, 본문 텍스트는 **모두 동일한 좌측 시작 x좌표**를 가져야 한다.

```
❌ 나쁜 예: 카드 1 헤드라인 x=0.65, 구분선 x=0.70, 본문 x=0.65 (3개가 어긋남)
✅ 원칙: 모든 카드 내부 요소는 카드 좌측 + 패딩(예: 0.20") 으로 통일된 x좌표 사용
```

**[AP-08] ACC 도형 line(테두리) 정책 통일**

ACC(액센트) 색상으로 채워진 도형(원형 숫자, 큰 강조 박스 등)은 한 제안서 내에서 line 속성이 통일되어야 한다. 한 슬라이드에서는 테두리 없음, 다른 슬라이드에서는 테두리 있음 → 시각적 일관성 붕괴.

```
❌ 나쁜 예: S6의 ACC 원형은 테두리 있음, S5의 ACC 원형은 테두리 없음
✅ 원칙: ACC 색상 도형의 line 속성을 컴포넌트 단위로 한 곳에서 정의하고, 모든 슬라이드에서 동일하게 사용
   기본값: line: { type: 'none' } — 테두리 없이 fill만 사용
```

### ★ 여백(white-space) 정량 판단 기준 — 가장 중요

여백 관리는 본 스킬에서 가장 빈번하게 실패하는 영역이다. "여백이 많아 보인다"는 주관 판단을 줄이기 위해 아래 정량 기준을 따른다.

#### 1) 슬라이드 레벨 (Content Box 전체)

| 측정 | 정의 | 기준 |
|------|------|------|
| **Slide Fill Rate** | 콘텐츠 도형/텍스트 점유 면적 ÷ Content Box 면적(10.63 × 5.91 = **62.82 in²**) | **≥ 65%** |
| **연속 빈 영역(Empty Patch)** | 콘텐츠가 없는 사각형 영역의 최대 크기 | **≤ 8 in² (가로 4″ × 세로 2″ 이내)** — 이보다 큰 빈 패치가 있으면 위반 |
| **하단 마진** | 마지막 콘텐츠 하단 ~ Content Box 하단(y=7.68) | **≤ 0.6″** (insight box 미사용 시) |

#### 2) 카드/그룹 레벨 (개별 카드 내부)

| 측정 | 정의 | 기준 |
|------|------|------|
| **Card Fill Rate** | 카드 내 텍스트+아이콘+장식 점유 면적 ÷ 카드 전체 면적 | **≥ 50%** |
| **수직 여백 비율** | 카드 내 텍스트 블록 아래 빈 공간 ÷ 카드 높이 | **≤ 25%** |
| **상단 패딩** | 카드 상단 ~ 첫 텍스트 | 카드 높이의 8~12% (그리드 내 일관) |

#### 3) 여백 위반 시 대응 우선순위 (반드시 1번부터 순서대로)

1. **콘텐츠 보강** (Step 0 자료 추가 추출 — 가장 우선)
2. **폰트 확대** (헤드라인 +2pt, 본문 +1~2pt — 단, 카드 그리드 내에서는 일괄 적용)
3. **도식화** (아이콘, 구분선, 컬러 바, 작은 차트, 진행률 바 추가)
4. **이미지/플레이스홀더 삽입** (관련 이미지 영역 표시)
5. **카드 수 축소** (4개 → 3개로 줄이고 카드 폭 확대)
6. **금지**: 의미 없는 장식 도형으로 빈 공간을 채우는 것 (AP-03)

### ★ Fill-Rate 규칙 + 카드 내부 밀도 (매우 중요)

**슬라이드 레벨**: 콘텐츠 영역 채움률 65% 이상 유지 (위 정량 기준 적용)

**카드/블록 레벨 (더 중요)**: 개별 카드 내부의 여백이 50%를 넘지 않도록 관리한다. 카드 안에 텍스트가 2~3줄뿐이라면 **반드시** 아래 방법 중 하나 이상으로 밀도를 높인다:

**밀도 확보 방법 (우선순위순):**

1. **폰트 크기 확대**: 카드 내 핵심 텍스트를 키워 시각적 무게감 확보
   - 카드 타이틀: 최소 14pt, 권장 16~20pt
   - 카드 본문: 최소 11pt, 권장 12~14pt
   - 핵심 수치: 28pt+ 대형 타이포그래피 적극 활용
   - **콘텐츠가 부족하면 폰트를 키워서 공간을 채운다** (빈 공간 방치 금지)

   **폰트 자동 확대 룰 (카드 그리드 내):**
   - 카드 본문이 **3줄 미만**이면 본문 폰트 +2pt 적용
   - 카드 본문이 **1줄**이면 본문 폰트 +4pt 적용
   - 단, **카드 그리드 내에서는 가장 긴 카드 기준으로 폰트를 결정하고 모든 카드에 동일하게 적용** (AP-06 준수)
   - 짧은 카드는 폰트가 같은 채로 빈 공간이 생기면 → 콘텐츠 보강 또는 도식화로 채움

2. **아이콘 확대 배치**: 카드 좌측이나 상단에 관련 아이콘을 48~64px로 크게 배치

3. **배경 이미지 + 오버레이 기법** (적극 권장):
   ```
   ┌────────────────────────────┐
   │  [배경 이미지 전체 깔기]      │  ← 카드 영역 전체에 관련 이미지
   │  ┌──────────────────────┐  │
   │  │ 그라데이션 오버레이     │  │  ← 반투명 도형 (좌→우 또는 하→상)
   │  │ opacity: 0.7~0.85     │  │     투명→불투명 그라데이션
   │  │                       │  │
   │  │  텍스트 (불투명 영역)   │  │  ← 텍스트는 오버레이의 불투명 부분에
   │  └──────────────────────┘  │
   └────────────────────────────┘
   ```
   - 이미지와 텍스트는 독립적으로 존재할 필요 없음
   - 이미지 위에 그라데이션 오버레이를 깔고 그 위에 텍스트를 올리면
     한 공간에 이미지와 텍스트를 모두 활용하는 효과
   - 이미지가 필요한 부분에는 **`[IMAGE: 설명]`** 플레이스홀더를 코드 주석으로 표시

4. **장식 요소 추가**: 구분선, 컬러 바, 배경 도형 등으로 시각적 밀도 확보

### ★ Insight Box 정책 (적재적소 원칙 — 모든 페이지 강제 배치 금지)

> **★ 정책 변경**: 이전에는 모든 본문 페이지 하단에 Insight Box를 의무 배치했지만, 이제는 **강조가 필요한 페이지에만 적재적소로 사용**한다. 모든 페이지에 동일한 구조의 요약 박스가 들어가면 제안서가 단조로워지고, 박스의 의미가 희석된다.

**Insight Box를 사용해야 하는 페이지 유형:**
- 우리의 핵심 제안 / 솔루션 선언 페이지
- 우리의 역량·강점·차별점 페이지
- 핵심 경험·실적·레퍼런스 페이지
- 수치 KPI 강조 페이지의 결론
- 챕터의 마지막 페이지(챕터 요약)

**Insight Box를 사용하면 안 되는 페이지:**
- 단순 정보 나열 페이지 (목차, 회사 소개, 일정 등)
- 강조 페이지(이미 다크 배경/대형 타이포로 충분히 강조됨)
- 같은 챕터에서 이미 insight box가 등장한 페이지 (한 챕터 내 1~2개 제한)

**Insight Box 규격 (사용 시):**

```javascript
// Insight Box 규격 — 강조용으로 사용하므로 폰트와 높이를 키움
const INSIGHT_Y = CB_Y + CB_H - insightH;  // Content Box 하단에 배치
const INSIGHT_H = 0.60;  // 최소 높이 0.60" (이전 0.50"에서 격상)
const INSIGHT_FONT = 14; // 최소 14pt (이전 12pt에서 격상) — 핵심 결론이면 16pt까지 허용
```

**Insight Box 규칙:**
1. **최소 높이 0.60″**: 폰트가 커진 만큼 박스 높이도 격상
2. **최소 폰트 14pt** (핵심 결론이면 16pt): 강조 역할이므로 본문(12pt)보다 확실히 크게
3. **겹침 방지 필수**: 위의 카드/콘텐츠와 최소 0.15″ 간격 확보. 콘텐츠 높이를 계산하여 insight box가 겹치지 않도록 y좌표를 동적 조정
4. **Content Box 경계 내 배치**: insight box의 하단이 CB_Y + CB_H(=7.68″)를 넘지 않도록
5. **한 챕터 내 1~2개 제한**: 너무 자주 등장하면 강조 효과 상실

```
❌ 나쁜 예: 모든 본문 페이지 하단에 동일한 구조의 insight box → 단조로움
✅ 좋은 예: 챕터당 1~2개, 핵심 메시지가 있는 페이지에만 적재적소로 배치
```

### ★ pptxgenjs 꾸밈효과 적극 활용 가이드

pptxgenjs에서 지원하는 시각 효과를 적극적으로 활용하여 디자인 퀄리티를 높인다.

**1. 그림자 (Shadow)** — 카드, 아이콘, 주요 도형에 적용:
```javascript
sl.addShape(pptx.shapes.ROUNDED_RECTANGLE, {
  shadow: {
    type: 'outer',
    color: '000000',
    blur: 10,
    offset: 3,
    angle: 315,
    opacity: 0.2
  }
});
```

**2. 투명도 (Transparency)** — 오버레이, 배경 도형에 적용:
```javascript
// 반투명 오버레이 (이미지 위에 텍스트 배치 시)
sl.addShape(pptx.shapes.RECTANGLE, {
  fill: { color: DOM, transparency: 25 },  // 75% 불투명
  x: 0, y: 0, w: '100%', h: '100%'
});
```

**3. 그라데이션 (Gradient Fill)** — 배경, 오버레이, 카드에 적용:
```javascript
sl.addShape(pptx.shapes.RECTANGLE, {
  fill: {
    type: 'gradient',
    stops: [
      { position: 0, color: DOM, transparency: 0 },
      { position: 100, color: DOM, transparency: 80 }
    ],
    direction: 'right'  // 좌→우 그라데이션 (좌측 불투명, 우측 투명)
  }
});
```

**4. 둥근 모서리 (Rounded Rectangle)** — 카드에 적용:
```javascript
sl.addShape(pptx.shapes.ROUNDED_RECTANGLE, {
  rectRadius: 0.15,  // 모서리 둥글기 (인치)
  fill: { color: CD },
  shadow: { type: 'outer', blur: 8, offset: 2, opacity: 0.15 }
});
```

**5. 이미지 플레이스홀더 표기 규칙**:
- 이미지가 들어가야 하는 위치에는 코드 주석으로 `// [IMAGE: 이스포츠 경기장 전경]` 형태로 표시
- 실제 이미지 파일이 없으므로, 해당 영역에는 SEC 또는 LT 색상의 플레이스홀더 도형을 배치
- 플레이스홀더 도형 중앙에 작은 텍스트로 "이미지 영역" 표시
```javascript
// [IMAGE: 이스포츠 경기장 전경]
sl.addShape(pptx.shapes.RECTANGLE, {
  x: cardX, y: cardY, w: cardW, h: cardH,
  fill: { color: LT }
});
sl.addText('📷 이미지 영역', {
  x: cardX, y: cardY + cardH/2 - 0.15, w: cardW, h: 0.3,
  fontSize: 9, color: TG, align: 'center'
});
```

## Phase 3.5: 자체 검수 (PPT 생성 직후, 사용자 전달 전 — 필수)

PPT 코드를 생성한 직후, 사용자에게 결과물을 전달하기 전에 **반드시** 아래 체크리스트를 코드 레벨에서 검수하고 문제가 있으면 수정 후 재생성한다. 이 단계를 건너뛰지 않는다.

### 검수 체크리스트

**1. 요소 겹침/정렬 검사**
- 장식 도형(바, 원, 선 등)이 텍스트와 겹쳐 가독성을 해치지 않는가?
- 간지의 장식 요소가 챕터 라벨/타이틀 텍스트 영역(x=0.57~8.77)과 물리적으로 겹치지 않는가?
- 모든 텍스트 요소가 슬라이드 경계(11.69 × 8.27) 안에 있는가?

**2. 색상 대비/가독성 검사 (WCAG AA 기준)**
- 어두운 배경(DARK_BG, DOM, CB) 위의 텍스트가 밝은 색(W, ACC, TF)인가?
- 밝은 배경(W, LT) 위의 텍스트가 어두운 색(DOM, 333355)인가?
- **대비 방향 검사**: 밝은 배경 + 어두운 대면적 요소(카드, 블록)가 없는가? (AP-01 위반)
- **대비 비율 검사**: 텍스트와 배경의 명도 대비가 WCAG AA 기준(4.5:1) 이상인가? 특히 배너/오버레이 위 텍스트에서 투명도로 인해 대비가 낮아지는 경우를 주의
- 아이콘/이모지가 배경과 동일 톤이라 보이지 않는 경우가 없는가? (다크 카드 안의 어두운 아이콘 등)
- 다크 카드 안에 아이콘을 배치할 경우, 아이콘 배경에 밝은 색 원형(OVAL)을 깔아 가독성을 확보했는가?

**3. 분류별 시각 차별화 검사**
- 표지/마무리 vs 간지: 배경색이 확실히 다른가? (동일 색상이면 수정 필수)
- 본문-강조 vs 본문-기본: 강조 페이지가 배경 반전, 대형 블록 등으로 시각적으로 확연히 다른가?
- 강조 페이지가 콘텐츠 클라이맥스(핵심 제안/솔루션 선언, 비전 제시)에 배치되어 있는가?
- 강조 페이지에 레퍼런스 갤러리, 단순 테이블 같은 보조 콘텐츠가 배치되어 있지 않은가?

**4. 폰트 규격 검사**
- 모든 본문 페이지의 핵심 메시지(hdr 함수의 msg)가 `FN_XB` (Pretendard ExtraBold)로 적용되어 있는가?
- 표지 메인카피, 간지 타이틀, 마무리 다짐카피가 모두 `FN_XB`인가?

**5. 콘텐츠 깊이 검사 (가장 중요)**
- 각 카드/블록에 제목 + 본문 2줄 이상 + 수치/근거가 포함되어 있는가?
- 콘텐츠가 부족하여 아이콘/장식 도형으로 빈 공간을 메우고 있지는 않은가?
- 텍스트가 2줄 이하인 카드가 있다면, 콘텐츠를 보강할 수 있는지 먼저 검토했는가?
- 빈 공간의 원인이 "디자인 문제"가 아니라 "콘텐츠 부족"은 아닌가?

**6. 카드 내부 밀도 검사 (콘텐츠가 충분한 전제 하에)**
- 개별 카드/블록 내부의 여백이 50%를 넘지 않는가?
- 콘텐츠는 충분하나 폰트가 너무 작아 비어 보이는 경우: 폰트 확대
- 콘텐츠가 충분하고 폰트도 적절한데 여전히 빈 경우: 아이콘/수치 강조 추가
- 카드 타이틀이 16pt 이상인가? 카드 본문이 12pt 이상인가?
- 슬라이드 전체 콘텐츠 영역 채움률이 60% 이상인가?

**7. Content Box 준수 검사**
- 모든 본문 페이지의 콘텐츠가 Content Box 경계(x=0.53~11.16", y=1.77~7.68") 안에 배치되어 있는가?
- Content Box 내부 패딩(0.15")이 지켜지고 있는가?
- 카드/아이콘 등 반복 요소가 Content Box 내에서 균등 간격으로 배치되어 있는가?
- 서로 다른 본문 페이지 간에 콘텐츠 시작점/끝점이 Content Box 기준으로 정렬되어 통일성이 있는가?

**8. 헤더 일관성 검사**
- 모든 본문 페이지(기본+강조)의 헤더 좌표가 동일한가? (x=0.47, y=0.21/0.68/1.18/1.55)
- 강조 페이지에서도 헤더가 생략되지 않았는가?

**9. 안티패턴 위반 검사 (AP-01~08)**
- [AP-01] 본문 페이지 콘텐츠 영역 내 단일 다크 블록 면적이 25%를 초과하는가? → 배경을 다크로 전환하거나 다크 블록을 ACC 라인 + 밝은 카드로 대체
- [AP-02] 상하 하드 컬러 분할(50:50)이 있는가? → 비율 조정(70:30) 또는 그라데이션 전환 적용
- [AP-03] 내부가 빈 장식 도형(원형, 사각형)이 있는가? → 아이콘, 번호, 라벨 텍스트 삽입
- [AP-04] 배너/히어로 영역에 텍스트만 있고 시각 요소가 없는가? → 아이콘 또는 장식 추가
- [AP-05] 텍스트박스 width/height 부족으로 텍스트가 도형 밖으로 빠져나가거나 강제 줄바꿈되는 곳이 있는가? → textbox 크기 재계산, autoFit 적용
- [AP-06] 카드 그리드 내 카드들의 헤드라인/본문 폰트 크기가 모두 동일한가? → 가장 긴 카드 기준으로 통일
- [AP-07] 카드 그리드 내 헤드라인/구분선/본문의 좌측 시작 x좌표가 모두 같은가? → 카드 좌측 + 패딩으로 통일
- [AP-08] ACC 색상 도형의 line(테두리) 속성이 모든 슬라이드에서 동일한가? → 컴포넌트 단위로 통일 (기본: line none)

**10. 여백 정량 기준 검사**
- 슬라이드 Fill Rate가 65% 이상인가?
- 연속 빈 영역(Empty Patch)이 8 in² (4″ × 2″) 이하인가?
- 각 카드의 Card Fill Rate가 50% 이상인가?
- 카드 내 수직 여백 비율이 25% 이하인가?
- 위반 시: 콘텐츠 보강 → 폰트 확대 → 도식화 → 이미지 → 카드 수 축소 순으로 대응

**11. 간지 / 강조 페이지 다양성 검사**
- 한 제안서 내 간지가 2개 이상이면 서로 다른 type(A/B/C/D)을 사용하는가?
- 연속된 두 간지가 동일 type을 사용하지 않는가?
- 한 제안서 내 강조 페이지가 2개 이상이면 서로 다른 variant(dark-keywords / image-overlay / split-impact)를 사용하는가?

**12. Insight Box / 마무리 페이지 검사**
- Insight Box가 모든 본문 페이지에 의무 배치되어 있지 않은가? (적재적소 원칙)
- Insight Box가 강조 가치가 있는 페이지(제안/역량/강점/경험/KPI 결론)에만 사용되었는가?
- Insight Box 폰트가 14pt 이상인가? (이전 12pt에서 격상)
- Insight Box 높이가 0.60″ 이상인가?
- Insight Box가 위의 콘텐츠와 겹치지 않는가? (최소 0.15″ 간격)
- 한 챕터 내 Insight Box가 1~2개 이내인가?
- 마무리 페이지의 히어로가 약속/다짐 카피인가? ("감사합니다"가 히어로가 아닌가?)
- 마무리 히어로 텍스트박스 width가 7″ 이상이고, 글자수 × 폰트사이즈 계산값을 수용하는가? (세로 1자씩 배치 방지)

**자동 실행 절차 (생략 금지):**

```
1. PPTX → PNG 변환 (PowerShell + PowerPoint COM, Windows 기본)
2. Read 툴로 모든 PNG를 (병렬로) 읽어들여 Claude vision으로 직접 분석
3. 슬라이드별 점수·이슈·개선사항 표 작성
4. Critical 이슈(텍스트 잘림, AP 위반 등) 발견 시 수정 코드 생성 → 재생성
5. 최종 보고를 사용자에게 전달
```

### Step 1 — PPTX → PNG 변환 (Windows / PowerPoint COM)

LibreOffice가 없어도 PowerPoint가 설치된 Windows 환경이면 아래 PowerShell 스크립트로 변환한다. `slides_png/` 폴더에 `slide_01.png ... slide_NN.png` 생성.

```powershell
$pptPath = "절대경로\file.pptx"
$outDir  = "절대경로\slides_png"
New-Item -ItemType Directory -Force -Path $outDir | Out-Null
$ppt = New-Object -ComObject PowerPoint.Application
$ppt.Visible = [Microsoft.Office.Core.MsoTriState]::msoTrue
$pres = $ppt.Presentations.Open($pptPath, $true, $false, $false)
$i = 1
foreach ($slide in $pres.Slides) {
  $name = "slide_{0:D2}.png" -f $i
  $slide.Export("$outDir\$name", "PNG", 1600, 1131)
  $i++
}
$pres.Close(); $ppt.Quit()
"DONE: $($i-1) slides"
```

> **PowerPoint도 없을 때 폴백**: LibreOffice → `soffice --headless --convert-to pdf` 후 `pdftoppm`. 둘 다 없으면 사용자에게 PPTX를 직접 열어 스크린샷을 요청. **절대 Phase 4를 스킵하지 않는다.**

### Step 2 — Claude 직접 vision 분석

PNG가 생성되면 Read 툴을 4장씩 병렬 호출하여 모든 슬라이드를 읽어들인다. 각 슬라이드에 대해 다음 6개 축으로 평가한다:

| 축 | 체크 항목 |
|----|---------|
| **alignment** | Content Box 경계 준수, 카드 정렬, 헤더 좌표 일관성 |
| **fillRate** | 슬라이드 채움률 60%+, 카드 내부 여백 50% 이하 |
| **color** | WCAG AA 4.5:1 대비, 팔레트 일관성 |
| **typography** | 헤드라인 ExtraBold, 본문 위계, 텍스트 잘림 여부 |
| **visuals** | 빈 장식 도형(AP-03), 배너 아이콘(AP-04) 검사 |
| **antipattern** | AP-01(밝은 배경+다크 카드 25%+), AP-02(hard split), 간지/강조 페이지 변형 중복 |

### Step 3 — 보고서 형식

```
[시각 QA 결과 — 자동 분석]
종합 점수: XX/100

| # | 분류 | 점수 | 핵심 이슈 |
|---|------|-----|---------|
| 1 | 표지 | 85 | ... |
| ... |

주요 이슈 (우선순위):
1. [Critical] 슬라이드 N — 텍스트 잘림 → 즉시 수정
2. [High]     슬라이드 N — AP-01 위반 → 수정 권장
...
```

### Step 4 — 자동 수정 트리거

- **Critical** (텍스트 잘림, 슬라이드 경계 이탈, 헤더 누락): 사용자 확인 없이 즉시 수정 → 재생성 → 재QA
- **High** (AP 위반, 70점 미만): 수정안 제시 후 사용자 확인
- **Medium/Low**: 보고서에만 기록

## 환경 설정

이 스킬을 처음 사용할 때 아래 사항을 확인하세요:

| 항목 | 설정 방법 |
|------|----------|
| **ANTHROPIC_API_KEY** | 환경변수로 설정. 미설정 시 카피 정제/시각 QA 단계를 스킵하고 수동 진행 |
| **Node.js 패키지** | 작업 디렉토리에서 `npm install pptxgenjs sharp @anthropic-ai/sdk adm-zip jszip` |
| **Pretendard 폰트** | 시스템에 설치 필요. 미설치 시 Arial로 대체 |
| **LibreOffice** (선택) | 시각 QA의 PPTX→이미지 변환에 필요. 없으면 QA 단계 스킵 |
