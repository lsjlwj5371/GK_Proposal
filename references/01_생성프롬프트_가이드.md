## 1. 전체 워크플로우 개요

```
[클라이언트 분석 자료]
        |
        v
  STEP 1: 자료 정리
  - 사업 개요 추출
  - 핵심 메시지 도출
  - 데이터 분류
  - 제안 대상 브랜드 색상 파악
        |
        v
  STEP 1.5: 제안서 논리 구조 설계
  - 제안서의 스토리라인을 먼저 설계
  - 각 챕터·페이지의 역할과 논리적 흐름 확정
  - 이 단계에서는 템플릿을 고려하지 않음
        |
        v
  STEP 2: 콘텐츠 구조화 (논리 -> 템플릿 매핑)
  - 페이지 4분류 (표지/간지/본문-기본/본문-강조/마무리) 적용
  - 확정된 논리 구조에 맞는 템플릿 선택
  - 적합한 템플릿이 없으면 커스텀 슬라이드 설계
  - 콘텐츠 배분 + 시각적 밀도 점검
        |
        v
  STEP 3: 프롬프트 작성
  - 마스터 프롬프트 (헤더 좌표 규격 준수)
  - 슬라이드별 상세 지시
  - 시각적 밀도 Fill-Rate 40% 이하 빈 공간 확인
  - 생성 실행
        |
        v
  STEP 4: 검수/수정
  - 헤더 위치 일관성 확인 (모든 본문 페이지 동일)
  - 콘텐츠 영역 통일성 확인 (y=1.77 시작)
  - 빈 공간 비율 점검 (40% 초과 시 시각 요소 추가)
  - 콘텐츠 정확도
  - 최종 보정
```

## 2. STEP 1: 클라이언트 분석 자료 정리

### 2.1 필수 수집 정보

분석 자료에서 아래 항목을 반드시 추출합니다:

| 카테고리 | 필수 항목 | 예시 |
|---------|----------|------|
| **기본 정보** | 사업명, 발주처, 사업기간, 예산규모 | "2026 스마트시티 통합관제 구축" |
| **배경/현황** | 현재 과제, 시장 환경, 경쟁 상황 | "기존 수동 관제의 한계..." |
| **제안 내용** | 솔루션, 핵심 기능, 차별점 | "AI 기반 실시간 분석 플랫폼" |
| **실적** | 유사 프로젝트, KPI, 수상/인증 | "3년간 15개 지자체 구축" |
| **조직** | 투입 인력, 조직 구성, 핵심 멤버 | "PM 1, PL 2, 개발 5, QA 2" |
| **비용/일정** | 견적, 마일스톤, 납기 | "총 4.2억, 6개월" |
| **회사 정보** | 연혁, 비전, 파트너 | "2010년 설립, 글로벌 5개 거점" |
| **제안 대상 브랜드 색상** | 로고, 웹사이트, CI 가이드에서 추출 | "제네시스: #0B1628 다크네이비, #8B7355 쿠퍼" |

> **색상 수집은 STEP 1에서 반드시 수행합니다.**
> 제안 대상의 브랜드 색상이 파악되면 6:3:1 시스템으로 DOMINANT/SECONDARY/ACCENT를 결정합니다.
> 상세 프로세스: 섹션 5 '색상 결정 프로세스' 참조.

### 2.2 핵심 메시지 도출 프레임

분석 자료에서 **3가지 핵심 축**을 추출합니다:

```
1. WHY (왜 우리인가?)
   -> 표지 캐치프레이즈, 챕터 메시지에 활용
   -> 예: "15년 현장 경험 + AI 기술력"

2. WHAT (무엇을 제안하는가?)
   -> 솔루션/서비스 소개 슬라이드에 활용
   -> 예: "통합관제 플랫폼 + 현장 운영 체계"

3. HOW (어떻게 실행하는가?)
   -> 프로세스/조직/일정 슬라이드에 활용
   -> 예: "4단계 애자일 구축 + 전담 운영팀"
```

### 2.3 데이터 분류표

추출한 데이터를 아래 형태로 정리합니다:

```markdown
## 분석 데이터 요약

### 정량 데이터
- 매출: 150억 (2025)
- 직원수: 120명
- 프로젝트 수: 45건
- 만족도: 98.5%

### 정성 데이터
- 핵심 강점: AI 분석 기술력, 현장 운영 경험
- 차별점: 유일한 통합 플랫폼 보유사
- 비전: "기술로 더 안전한 도시를 만든다"

### 비주얼 자산
- 시스템 스크린샷: 3장
- 현장 사진: 5장
- 조직도 원본: 있음
- 로고: 파트너 8개사
```

## 3. STEP 1.5: 제안서 논리 구조 설계 (콘텐츠 우선 원칙)

### 3.1 원칙 선언

```
★ 제안서의 설득력 = 논리 구조의 탄탄함
  - 템플릿은 시각적 '그릇'이고, 콘텐츠는 '내용물'
  - 그릇에 내용물을 끼워맞추지 않는다
  - 내용물에 맞는 그릇을 고른다
  - 적합한 그릇이 없으면 새로 만든다
```

### 3.2 논리 구조 설계 프로세스

**STEP 1.5** (자료 정리 후, 템플릿 매핑 전):

1. **스토리라인 설계**: 제안서 전체의 서사 흐름 확정
   - 왜 이 제안을 하는가 (배경/동기)
   - 무엇이 문제인가 (과제/현황)
   - 어떻게 해결하는가 (솔루션/전략)
   - 왜 우리인가 (실적/차별점)
   - 어디까지 갈 수 있는가 (확장/비전)

2. **페이지별 역할 정의**: 각 페이지가 전체 논리에서 어떤 역할을 하는지 명확히
   ```
   예시:
   Page 3 -> "숫자로 실행력을 증명하여 신뢰 확보" (KPI + 프로젝트 하이라이트)
   Page 8 -> "세 가지 문제의 근본 원인을 하나로 수렴" (문제 종합 -> 해결책 전환)
   ```

3. **페이지 간 전환 논리**: 앞 페이지의 결론이 다음 페이지의 전제가 되는 흐름
   ```
   Problem A + B + C -> 공통 원인 -> 3개 Module로 동시 해결
   (이 논리 흐름이 먼저이고, 이에 맞는 슬라이드는 나중)
   ```

### 3.3 템플릿 매핑 판단 기준

| 상황 | 판단 | 조치 |
|------|------|------|
| 콘텐츠가 기존 템플릿에 **정확히** 맞음 | 템플릿 사용 | S번호 지정 |
| 콘텐츠가 템플릿에 **대략** 맞지만 항목 수 다름 | 템플릿 변형 | S번호 기반 커스텀 |
| 콘텐츠가 어떤 템플릿에도 **맞지 않음** | 커스텀 제작 | CUSTOM 지정 |
| 하나의 논리를 **2개 이상 템플릿에 분산**해야 함 | 커스텀 제작 | 하나의 통합 슬라이드 |

> 예: PDF 참고 자료의 8페이지 "Problem A + B" 슬라이드는
> 기존 S04(Before/After)에 맞지 않는 구조이므로 CUSTOM으로 제작.
> 단, 헤더 영역은 동일하게 유지.

### 3.4 목차 설계 원칙

> **핵심 원칙: 논리 구조 우선 (Content-First)**
> 제안서의 목차와 콘텐츠 구조는 반드시 **제안 내용의 논리적 흐름**에 따라 설계합니다.
> 템플릿 슬라이드에 내용을 끼워맞추는 것이 아니라, 탄탄한 논리 구조를 먼저 확정한 뒤
> 그에 맞는 템플릿을 선택합니다. 적합한 템플릿이 없으면 커스텀 슬라이드를 제작합니다.

**기본 구조** (5챕터 기준):
```
Ch1. 산업 이해 / 과제 진단    -> 왜 이 사업이 필요한가
Ch2. 솔루션 / 전략 제안       -> 무엇을 제안하는가
Ch3. 수행 체계 / 운영 계획    -> 어떻게 실행하는가
Ch4. 실적 / 회사 소개         -> 왜 우리를 선택해야 하는가
Ch5. 비용 / 일정              -> 얼마에, 언제까지
```

> 위는 참고 구조일 뿐입니다. 제안서의 핵심 메시지와 설득 논리에 따라 챕터 수와
> 순서는 자유롭게 변경합니다. 예: 7챕터(현대차 제안서)처럼 세분화도 가능합니다.

**목차 변형 패턴**:
| 제안서 유형 | 목차 구성 |
|------------|----------|
| 공공입찰 | 사업이해 -> 기술제안 -> 수행체계 -> 실적 -> 비용일정 |
| IT솔루션 | 현황분석 -> 솔루션소개 -> 구축계획 -> 레퍼런스 -> 견적 |
| 회사소개 | 비전 -> 사업영역 -> 핵심기술 -> 실적 -> 팀/연혁 |
| 사업계획 | 시장분석 -> 사업모델 -> 추진전략 -> 재무계획 -> 팀 |
| 파트너십 제안 | 실적증명 -> 과제진단 -> 솔루션(모듈별) -> 통합가치 -> 로드맵 |

## 4. STEP 2: 콘텐츠 구조화 (4분류 + 템플릿 매핑)

### 4.1 페이지 4분류 체계 (표지/간지/본문-기본/본문-강조/마무리)

> **제안서의 모든 페이지는 5가지 유형 중 하나로 분류됩니다.**
> (편의상 "4분류"로 표기하지만, 본문을 기본/강조로 세분하면 5유형입니다.)
> 각 유형은 고유한 시각적 무게감, 레이아웃 규칙, 배치 규칙을 갖습니다.

| 분류 | 역할 | 시각적 무게감 | 배경 | 헤더 영역 |
|------|------|-------------|------|----------|
| **표지 (Cover)** | 제안서 첫 페이지. 임팩트 카피 + 대표 이미지 | 최고 | 다크(DARK_BG) 또는 전면 이미지 | 없음 (자유 배치) |
| **간지 (Section Divider)** | 챕터 전환. 무게감 있는 카피 + 핵심 메시지 | 높음 | DOMINANT 또는 다크 | 없음 (자유 배치) |
| **본문-기본 (Standard)** | 일반 정보 전달 페이지 | 중간 | 흰색(#FFFFFF) | **헤더 좌표 고정** |
| **본문-강조 (Highlight)** | 챕터 내 핵심 메시지 전달. 시각적 임팩트 강화 | 중상 | 흰색 기본, 또는 옵션 배경 적용 가능 | **헤더 좌표 고정** |
| **마무리 (Ending)** | 제안서 마지막. 다짐/의지를 담은 카피 | 최고 | DOMINANT 또는 다크 | 없음 (자유 배치) |

### 4.2 각 분류 상세

#### 4.2.1 표지 (Cover) 상세

```
+----------------------------------------------------------------+
|                                                                |
|  [대표 이미지 영역 또는 다크 배경 + 장식 도형]                |
|                                                                |
|          +-----------------------------+                      |
|          |  임팩트 카피라이트 문구      |                      |
|          |  (Bold 30~36pt, 2줄 이내)    |                      |
|          +-----------------------------+                      |
|                                                                |
|  캐치프레이즈 키워드 (ACCENT 색상, 16pt)                      |
|  핵심 메시지 1줄 (10~12pt, 밝은 회색)                          |
|                                                                |
|  +----------+ +----------+ +----------+                       |
|  | Module A  | | Module B  | | Module C  |  <- 선택: 모듈 뱃지 |
|  +----------+ +----------+ +----------+                       |
|                                                                |
|  제안사 정보 | 제안 대상 | 날짜 | CONFIDENTIAL                |
+----------------------------------------------------------------+
```

**표지 필수 요소**:
- 임팩트 카피라이트: 제안서 전체를 관통하는 한마디 (예: "제네시스를 경험하는 모든 이동, 그라운드케이가 설계합니다")
- 대표 이미지 또는 시각 요소: 사진 플레이스홀더, 또는 다크배경 + 장식 도형(투명 원형 등)
- 캐치프레이즈 키워드: 3~4개 핵심 단어 (영문 권장)
- 제안사/발주처 정보: 하단 작은 글씨

**표지 디자인 원칙**:
- 텍스트는 3계층 이하: 메인 카피 + 서브 카피 + 정보
- 이미지가 있으면 텍스트와 겹치지 않는 영역에 배치
- 이미지가 없으면 장식 도형(투명 원형, 기하학 패턴)으로 시각적 깊이감 부여
- 색상은 다크 배경 + 밝은 텍스트 (또는 전면 이미지 + 오버레이)

#### 4.2.2 간지 (Section Divider) 상세

```
+----------------------------------------------------------------+
|  [DOMINANT 배경]                                                |
|                                                                |
|  CHAPTER (또는 Module 등)     <- 작은 라벨 (ACCENT, 11pt)      |
|  01                           <- 챕터 번호 (Bold, 50~56pt)     |
|  ----                         <- ACCENT 구분선                  |
|                                                                |
|  핵심 카피라이트 문구          <- 챕터 핵심 (Bold, 22~28pt)     |
|  (2줄까지 허용)                  이 부분이 무게감의 핵심        |
|                                                                |
|  보조 설명 1~2줄              <- 밝은 회색 (12pt)               |
|                                                                |
|                     +------------------+                       |
|                     |  장식 도형 영역   | <- 투명 바, 원형 등   |
|                     +------------------+                       |
+----------------------------------------------------------------+
```

**간지 필수 요소**:
- 챕터 라벨: "CHAPTER", "Module A" 등 (ACCENT 색상, 자간 넓힘)
- 챕터 번호: 01~07 등 대형 숫자 (50pt+, WHITE)
- ACCENT 구분선: 번호 아래 짧은 가로선
- **핵심 카피라이트 문구**: 해당 챕터에서 말하고 싶은 것을 한마디로 (22~28pt, WHITE, Bold)
  - 이것이 일반 본문 헤더와 차별화되는 핵심. 단순 챕터명이 아니라 **메시지성 카피**
  - 예: "처음 만나는 업체의 영업이 아닙니다" (O), "제안의 출발점" (X -- 이건 라벨)
- 보조 설명: 카피의 부연, 이 챕터에서 다루는 내용 (12pt, 밝은 회색)

**간지 디자인 원칙**:
- 표지와 유사한 무게감: DOMINANT 또는 다크 배경, 큰 타이포그래피
- 일반 본문보다 확연히 다른 시각적 인상 (배경색, 폰트 크기로 차별)
- 장식 도형으로 단조로움 방지: 우측 가장자리 투명 바, 모서리 투명 원형 등
- 하단에 얇은 ACCENT 라인을 추가하여 표지와의 통일감 부여

**간지 vs 표지 차이**:
| 항목 | 표지 | 간지 |
|------|------|------|
| 배경 | 다크(DARK_BG) | DOMINANT |
| 메인 카피 크기 | 30~36pt | 22~28pt |
| 챕터 번호 | 없음 | 있음 (50pt+) |
| 이미지/사진 | 있음 (대표 이미지) | 없음 (장식 도형만) |
| 하단 정보 | 제안사/발주처 | 없음 |

#### 4.2.3 본문 (Content) 상세 -- 기본 페이지 vs 강조 페이지

> 본문 페이지는 모두 **동일한 헤더 좌표 규격**을 따릅니다.
> 기본 페이지와 강조 페이지의 차이는 **콘텐츠 영역의 시각적 강도**에 있습니다.

**기본 페이지 (Standard Content)**

일반적인 정보 전달 페이지. 기존 템플릿(S04~S41)을 최대한 활용합니다.

```
+----------------------------------------------------------------+
|  03  제안의 출발점           <- 페이지번호 + 목차라벨 (고정)    |
|  핵심 메시지 1줄             <- y=0.68 (고정)                   |
|  부제목                      <- y=1.18 (고정)                   |
|  ========================    <- y=1.55 구분선 (고정)             |
|                                                                |
|  [콘텐츠 영역 y=1.77~7.68]                                    |
|  +------+  +------+  +------+                                 |
|  | 카드  |  | 카드  |  | 카드  |  <- 기존 템플릿 활용           |
|  |      |  |      |  |      |                                 |
|  +------+  +------+  +------+                                 |
|  [인사이트 박스]                <- 하단 요약                    |
|                                              페이지 번호 -> 03 |
+----------------------------------------------------------------+
```

**기본 페이지 특성**:
- 흰색 배경 (#FFFFFF)
- 헤더 영역: 표준 좌표 고정 (섹션 6 참조)
- 콘텐츠: 기존 템플릿 레이아웃 활용 (S04~S41)
- 시각 요소: 최소 2가지 비텍스트 요소 (아이콘, 구분선, 태그 등)
- Fill-Rate: 60~80%

**강조 페이지 (Highlight Content)**

해당 챕터에서 가장 중요한 메시지를 전달하는 페이지.
기본 페이지보다 **시각적 임팩트**가 강합니다.

```
+----------------------------------------------------------------+
|  03  제안의 출발점           <- 페이지번호 + 목차라벨 (고정)    |
|  핵심 메시지 1줄             <- y=0.68 (고정)                   |
|  부제목                      <- y=1.18 (고정)                   |
|  ========================    <- y=1.55 구분선 (고정)             |
|                                                                |
|  [콘텐츠 영역 -- 시각적 강도 UP]                               |
|  +----------------------------------------------------+       |
|  |  ## 다크 또는 DOMINANT 배경 블록 ##                  |       |
|  |  대형 KPI 숫자 + 수렴 인포그래픽                     |       |
|  |  또는 좌우 분할 (이미지 50% + 데이터 50%)           |       |
|  |  또는 프로세스 플로우 전체                            |       |
|  +----------------------------------------------------+       |
|  [인사이트 박스 -- 강조 색상]                                   |
|                                              페이지 번호 -> 03 |
+----------------------------------------------------------------+
```

**강조 페이지 특성**:
- **배경 옵션** (다음 중 하나 선택 -- 옵션이지 필수가 아님):
  - **기본**: 흰색 배경이지만, 콘텐츠 영역 내에 **다크/DOMINANT 배경 블록** 사용
  - **전면 배경 이미지**: 전면 배경 이미지 + 반투명 오버레이
  - **컬러 배경**: DOMINANT 또는 DARK_BG를 전체 슬라이드 배경으로 적용 (표준 흰색과 차별)
  - 핵심은 **시각적 임팩트 차별화** -- 방법은 자유
- **헤더 영역: 기본 페이지와 100% 동일한 좌표** (절대 변경 금지)
  - 배경이 달라져도 헤더 구조(페이지 번호 + 목차라벨 + 메시지 + 부제목 + 구분선)는 **같은 좌표**에 배치
  - 다크 배경 적용 시 텍스트 색상만 밝게 전환 (좌표는 불변)
- 콘텐츠: 기존 템플릿에 구애받지 않고 자유롭게 설계 (CUSTOM)
- 시각 요소: 3가지 이상 (이미지 영역, 대형 KPI, 인포그래픽, 다크 블록 등)
- Fill-Rate: 75~90% (기본 페이지보다 높은 충전율)

**강조 페이지 활용 예시**:
| 예시 | 설명 |
|------|------|
| KPI + 프로젝트 하이라이트 복합 | 상단 3개 대형 KPI + 하단 3개 프로젝트 카드 + 역량 태그 |
| 문제 수렴 인포그래픽 | 3개 문제 -> 화살표 -> 1개 공통 원인 -> 3개 솔루션 |
| Before/After + 수치 비교 | 좌우 색상 분할 + 구체적 수치 + 체인 플로우 |
| 통합 가치 공식 | 상단 4개 가치 카드 + 하단 다크 블록에 공식 바 |
| 고객 여정 전체 플로우 | 5단계 플로우 + 3대 변수 매트릭스 + 점수 원형 |

#### 4.2.4 마무리 (Ending) 상세

```
+----------------------------------------------------------------+
|  [DOMINANT 또는 다크 배경]                                      |
|                                                                |
|  (장식 원형)                                                   |
|                                                                |
|         회사 로고 또는 이름                                    |
|         ----                                                   |
|                                                                |
|         "다짐이 느껴지는 카피라이트"                           |
|         (Bold 26~30pt, 2줄 이내)                               |
|                                                                |
|         Module A / Module B / Module C                         |
|                                                                |
|         ----                                                   |
|         CONFIDENTIAL | (c) 2026 회사명                         |
|                                                                |
|                                        (장식 원형)             |
+----------------------------------------------------------------+
```

**마무리 필수 요소**:
- **다짐 카피라이트**: 표지 카피의 변주. "앞으로 이렇게 하겠다"는 의지
  - 표지: "제네시스를 경험하는 모든 이동, 그라운드케이가 설계합니다" (선언)
  - 마무리: "제네시스를 경험하는 모든 이동을 설계합니다" (다짐/의지)
- 회사명/로고: 중앙 배치
- 모듈/서비스 라벨: 제안 범위 요약 (1줄)
- CONFIDENTIAL 표기

**마무리 디자인 원칙**:
- 표지와 유사한 시각적 무게감이지만, 더 간결하고 여백이 많음
- 장식 도형으로 표지와의 시각적 연결감 (동일한 투명 원형 등)
- 텍스트는 2계층 이하: 다짐 카피 + 보조 정보

> **절대 규칙: 마무리 페이지는 "감사합니다"로 끝내지 않습니다.**
> 마무리는 반드시 **표지의 핵심 메시지를 반향하는 다짐/서약 카피**를 포함해야 합니다.
> "감사합니다"만 있는 엔딩은 제안서의 마지막 인상을 약하게 만듭니다.

**마무리 카피 좋은 예 vs 나쁜 예**:

| 구분 | 예시 | 판정 |
|------|------|------|
| **나쁜 예** | "감사합니다" | X -- 메시지 없음 |
| **나쁜 예** | "Thank You" | X -- 관행적, 임팩트 없음 |
| **나쁜 예** | "끝까지 읽어주셔서 감사합니다" | X -- 설득이 아닌 감사 |
| **좋은 예** | "펜타포트의 모든 이동을\n안전하고 즐겁게 설계하겠습니다" | O -- 표지 카피 반향 + 다짐 |
| **좋은 예** | "제네시스를 경험하는 모든 이동을\n설계합니다" | O -- 선언적 다짐 |
| **좋은 예** | "기술과 경험으로\n더 안전한 도시를 만들겠습니다" | O -- 비전 + 의지 |
| **좋은 예** | "당신의 도시를 24시간 지킵니다" | O -- 약속/서약형 |

### 4.3 강조 페이지 배치 규칙 + 비율 (2-12%)

```
★ 강조 페이지 배치 규칙 (반드시 준수):

1. 챕터당 최대 2장까지만 허용
2. 강조 페이지 2장이 연속 배치되지 않도록 한다
   -> 사이에 반드시 기본 페이지 1장 이상 삽입
3. 챕터의 첫 번째 또는 마지막 본문 페이지에 배치 권장
4. 챕터에 본문이 1~2장만 있으면 강조 페이지는 1장까지만

예시 (O): 간지 -> [강조] -> 기본 -> 기본 -> [강조] -> 간지
예시 (X): 간지 -> [강조] -> [강조] -> 기본 -> 간지  <- 강조 연속 불가
예시 (X): 간지 -> [강조] -> [강조] -> [강조] -> 간지  <- 3장 초과 불가
```

> **강조 페이지 비율 가이드 (레퍼런스 제안서 5건 분석 결과)**
> 전체 본문 페이지 중 강조 페이지 비율은 **5~10%가 이상적** (최대 12%).
> 강조가 많아지면 "모든 것이 강조 = 아무것도 강조되지 않음"이 됩니다.

### 4.4 제안서 기본 흐름 + 페이지 수 가이드

> 사용자의 특별한 요청이 없는 경우, 아래 흐름을 기본으로 구성합니다.

```
[표지]
  |
  v
[간지 Ch1] -> [본문: 기본] -> [본문: 강조] -> [본문: 기본] -> ...
  |
  v
[간지 Ch2] -> [본문: 기본] -> [본문: 기본] -> [본문: 기본] -> ...
  |
  v
[간지 Ch3] -> [본문: 기본] -> [본문: 기본] -> [본문: 기본] -> ...
  |
  v
... (반복) ...
  |
  v
[마무리]
```

**기본 흐름 규칙**:
1. 제안서는 반드시 **표지**로 시작
2. 각 챕터는 반드시 **간지**로 시작
3. 간지 다음에는 **본문 페이지** (기본 + 강조 혼합)
4. 강조 페이지는 챕터당 최대 2장, 비연속 배치
5. 제안서는 반드시 **마무리**로 종료 (다짐 카피 필수, "감사합니다" 금지)
6. 목차 페이지(S03)는 표지 직후 또는 생략 가능 (선택)

**페이지 수 가이드 (강조 비율 5~10% 반영)**:

| 제안서 규모 | 총 페이지 | 표지 | 간지 | 기본 본문 | 강조 본문 | 마무리 |
|------------|----------|------|------|----------|----------|--------|
| 소규모 (10~15p) | ~12 | 1 | 3 | 6~7 | 1 | 1 |
| 중규모 (15~25p) | ~22 | 1 | 5~7 | 12~14 | 1~2 | 1 |
| 대규모 (25~40p) | ~35 | 1 | 6~8 | 22~26 | 2~3 | 1 |

> **전체 본문 페이지 중 강조 페이지 비율은 5~10%가 이상적 (최대 12%)**
> 예: 중규모 22페이지 -> 본문 약 14장 -> 강조 1~2장 (7~14%)

### 4.5 다중 레이아웃 + 분류 내 일관성 원칙

> **각 분류(표지/간지/본문-기본/본문-강조/마무리) 내에서 여러 가지 레이아웃 변형이 가능합니다.**

BUT:

> **같은 분류에 속하는 페이지들은 반드시 일관된 스타일 원칙을 유지해야 합니다.**

| 분류 | 반드시 일관된 원칙 | 변형 가능 범위 |
|------|------------------|--------------|
| **표지** | 항상 다크배경 + 임팩트 카피 | 레이아웃, 이미지 유무, 모듈 뱃지 유무 |
| **간지** | 항상 DOMINANT 배경 + 챕터 카피 | 구성 배치, 장식 도형 위치, 카피 줄 수 |
| **본문-기본** | 항상 흰색 배경 + 고정 헤더 + 60~80% fill rate | 템플릿 종류, 카드 수, 레이아웃 패턴 |
| **본문-강조** | 고정 헤더 유지 + 75~90% fill rate + 시각적 강도 UP | 배경 옵션, 콘텐츠 구성, 다크 블록 사용 |
| **마무리** | 다크/DOMINANT 배경 + 다짐 카피 | 장식 도형, 라벨 구성, 색상 톤 |

**프롬프트 슬라이드 리스트 표기법**:
```
| 순번 | 분류 | 유형 | 제목/카피 |
|------|------|------|----------|
| 1 | 표지 | COVER | "제네시스를 경험하는 모든 이동..." |
| 2 | 간지 | DIVIDER-Ch1 | "처음 만나는 업체의 영업이..." |
| 3 | 본문-강조 | HIGHLIGHT | KPI + 프로젝트 하이라이트 |
| 4 | 본문-기본 | STANDARD(S05) | 브랜드 접점 3대 변수 |
| 5 | 간지 | DIVIDER-Ch2 | "이동하는 모든 순간이..." |
| 6 | 본문-기본 | STANDARD(S07) | 솔루션 상세 |
| 7 | 본문-강조 | HIGHLIGHT(CUSTOM) | 고객 여정 플로우 |
| ... | ... | ... | ... |
| N | 마무리 | ENDING | "제네시스를 경험하는 모든 이동을..." |
```

### 4.6 프롬프트 생성 시 4분류 자동 적용

> 자료를 분석하고 PPT 제작 프롬프트를 생성할 때, 아래 순서로 4분류를 자동 적용합니다.

**자동 적용 절차**:
1. 자료의 논리 구조를 분석하여 챕터 수를 결정
2. 표지 카피라이트 도출 (WHY 메시지 기반)
3. 각 챕터별 간지 카피라이트 도출 (챕터의 핵심 메시지)
4. 각 챕터 내 본문 페이지를 기본/강조로 분류:
   - 해당 챕터에서 가장 핵심적인 1~2개 페이지를 **강조**로 지정
   - 나머지를 **기본**으로 지정
   - 강조 페이지가 연속되지 않도록 순서 조정
   - 전체 본문 중 강조 비율이 5~10% 범위인지 확인
5. 마무리 카피라이트 도출 (표지 카피의 다짐형 변주 -- "감사합니다" 금지)

## 5. 색상 결정 프로세스 (6:3:1 시스템 + 파생 색상)

> **기존 방식**: 그라운드케이의 고정 색상(#2E3092, #7B8CDE) 사용
> **변경된 방식**: 제안서를 받는 업체/대상의 브랜드 색상을 기반으로 결정

### 5.1 색상 결정 3단계

**1단계: 제안 대상 브랜드 색상 파악**
- 제안 대상 업체의 공식 브랜드 가이드라인, 로고, 웹사이트에서 핵심 색상 추출
- 명확한 브랜드 색상이 있는 경우: 그 색상을 DOMINANT로 채택
- 명확한 색상이 없는 경우: 업계/분야에 어울리는 색상 3개를 선택

**2단계: 6:3:1 비율로 3색 구성**

| 역할 | 비율 | 용도 | 선택 기준 |
|------|------|------|----------|
| **DOMINANT (주조색)** | **60%** | 배경, 간지 배경, 넓은 면적 | 제안 대상의 핵심 브랜드 색상, 또는 그것의 어두운 변주 |
| **SECONDARY (보조색)** | **30%** | 카드 배경, 헤더 바, 태그 | DOMINANT와 조화로운 중간 톤 (밝은 변주 또는 보색 계열) |
| **ACCENT (강조색)** | **10%** | 핵심 수치, CTA, 하이라이트 | 시선을 끄는 포인트 색상 (DOMINANT와 대비되는 밝은 색) |

**3단계: 파생 색상 자동 생성**

위 3색이 결정되면 아래 파생 색상을 자동 산출합니다:

| 파생 색상 | 생성 방법 | 용도 |
|----------|----------|------|
| DARK_BG | DOMINANT를 더 어둡게 (명도 -40~60%) | 표지/마무리 배경 |
| LIGHT_BG | SECONDARY의 매우 밝은 변주 (명도 95%) | 카드 배경, 밝은 영역 |
| CARD_BG | SECONDARY의 밝은 변주 (명도 90%) | 카드, 테이블 교차행 |
| SUBTLE | ACCENT의 매우 밝은 변주 (명도 93%) | 인사이트 박스, 태그 배경 |
| TEXT_DARK | #1A1A2E 또는 DOMINANT 기반 진한 색 | 본문 텍스트 |
| TEXT_GRAY | #6B7280 | 보조 텍스트, 설명 |
| TEXT_FAINT | SECONDARY의 밝은 변주 | 표지/간지 보조 텍스트 |

### 5.2 파생 색상 산출 규칙 (HSL 기반)

```
파생 색상 산출 규칙:
- DARK_BG: DOMINANT를 HSL로 변환, L을 10~15로 낮춤 (예: #0D0D2B)
- LIGHT_BG: DOMINANT를 HSL로 변환, L을 95~97로 높임 (예: #F0F0FF)
- CARD_BG: SECONDARY를 HSL로 변환, S를 20~30, L을 94~96 (예: #F0F2F5)
- SUBTLE: DOMINANT를 HSL로 변환, S를 15~25, L을 92~96 (예: #ECEDF5)
- TEXT_FAINT: DOMINANT를 기반으로 S를 낮추고 L을 60~70 (예: #8080A0)
```

### 5.3 색상 적용 예시

## 6. 헤더 영역 정밀 좌표 규격 (절대 고정)

> **이 섹션은 PPT 생성 시 가장 중요한 규격입니다.**
> 모든 본문 슬라이드에서 헤더 3요소의 위치가 1px이라도 어긋나면
> 전체 제안서의 일관성이 깨집니다.
> 표지, 간지, 마무리에는 이 헤더가 적용되지 않습니다 (자유 배치).

### 6.1 헤더 영역 좌표 (절대값 -- 인치 기준)

```
+------------------------------------------------------------------------+
|                                                                        |
|  x=0.47  페이지번호 + 목차라벨       y=0.21  h=0.25                   |
|          (예: "03  구조적 과제")                                        |
|                                                                        |
|  x=0.47  핵심 메시지 (ExtraBold, 16pt) y=0.68  h=0.47  w=10.75        |
|          (예: "현장에서 발견한 3가지 구조적 과제")                      |
|                                                                        |
|  x=0.47  부제목 (Regular, 10pt)      y=1.18  h=0.29  w=10.75         |
|          (예: "구조의 문제이기 때문에 해법도 구조를 바꿔야...")          |
|                                                                        |
|  ------- 구분선 (DOM) ------------  y=1.55  h=0.015 ----------       |
|                                                                        |
|  ~~~~~~~~ 호흡 여백 (0.22in) ~~~~~~~~                                  |
|                                                                        |
|  ======= 콘텐츠 시작선 ============  y=1.77  =================       |
|                                                                        |
|  [콘텐츠 영역]                                                         |
|  x=0.53 ~ 11.16 (w=10.63)                                             |
|  y=1.77 ~ 7.68 (h=5.91)                                               |
|  면적 = 10.63 x 5.91 = 62.82 sq.in                                    |
|                                                                        |
|  ------- 페이지 번호 -------------- y=7.68  (우측 하단) ------        |
|                                                                        |
+------------------------------------------------------------------------+
```

**핵심 정리**:
- **콘텐츠 시작선: y=1.77in** (구분선 y=1.55 + 호흡 여백 0.22in)
- **콘텐츠 영역**: x=0.53 ~ 11.16in (w=10.63in), y=1.77 ~ 7.68in (h=5.91in)
- **콘텐츠 영역 면적**: 10.63 x 5.91 = 62.82 sq.in

| 요소 | x (in) | y (in) | w (in) | h (in) | x (cm) | y (cm) |

| 핵심 메시지 | 0.47 | 0.68 | 10.75 | 0.47 | 1.19 | 1.73 |

| 구분선 | 0.47 | 1.55 | 10.75 | 0.015 | 1.19 | 3.94 |
| **콘텐츠 시작** | **0.53** | **1.77** | **10.63** | **5.91** | **1.35** | **4.50** |
| 하단 여백선 | -- | 7.68 | -- | -- | -- | 19.51 |
| 하단 페이지 번호 | 10.50 | 7.68 | 0.70 | 0.30 | 26.67 | 19.51 |

### 6.3 pptxgenjs 코드 스니펫

```javascript
function addStandardHeader(slide, pageNum, tocLabel, headerMsg, subMsg) {
  // 1. 페이지번호 + 목차라벨 (좌측 상단)
  slide.addText([
    { text: pageNum, options: { fontSize: 10, color: DOMINANT, bold: true, fontFace: 'Pretendard' } },
    { text: '  ' + tocLabel, options: { fontSize: 9, color: '6B7280', fontFace: 'Pretendard' } }
  ], { x: 0.47, y: 0.21, w: 5.0, h: 0.25, margin: 0 });

  // 2. 핵심 메시지 (ExtraBold 16pt)
  slide.addText(headerMsg, {
    x: 0.47, y: 0.68, w: 10.75, h: 0.47,
    fontSize: 16, fontFace: 'Pretendard ExtraBold', color: '1A1A2E', margin: 0
  });

  // 3. 부제목 (Regular 10pt)
  slide.addText(subMsg, {
    x: 0.47, y: 1.18, w: 10.75, h: 0.29,
    fontSize: 10, fontFace: 'Pretendard', color: '6B7280', margin: 0
  });

  // 4. 구분선 (DOM — brand primary)
  slide.addShape(pres.shapes.RECTANGLE, {
    x: 0.47, y: 1.55, w: 10.75, h: 0.015, fill: { color: DOMINANT }
  });

  // 5. 하단 페이지 번호 (우측)
  slide.addText(pageNum, {
    x: 10.50, y: 7.68, w: 0.70, h: 0.30,
    fontSize: 10, fontFace: 'Pretendard', color: '6B7280', align: 'right', margin: 0
  });
}

// ★ 콘텐츠는 반드시 y=1.77 이하에 배치
// ★ 콘텐츠 영역: x=0.53 ~ 11.16 (w=10.63), y=1.77 ~ 7.68 (h=5.91)
// ★ 콘텐츠 영역 면적: 10.63 x 5.91 = 62.82 sq.in
```

## 7. 시각적 밀도 규칙 (Fill-Rate) + 시각적 풍성함 가이드

### 7.1 콘텐츠 영역 정의

모든 본문 슬라이드에서 콘텐츠가 배치되는 영역:
- **시작점**: x=0.53in, y=1.77in
- **끝점**: x=11.16in, y=7.68in
- **크기**: w=10.63in x h=5.91in = **62.82 sq.in**

### 7.2 Fill-Rate 계산법 + 판정표

```
Fill-Rate = (콘텐츠가 차지하는 면적) / (전체 콘텐츠 영역 면적) x 100%

목표: 60% 이상 (빈 공간 40% 이하)
이상적: 65~80%
```

| Fill-Rate | 판정 | 조치 |
|-----------|------|------|
| **80%+** | 과밀 | 콘텐츠 분할 또는 폰트 축소 |
| **60~80%** | 적정 | 이상적 밸런스 |
| **40~60%** | 주의 | 보조 시각 요소 추가 검토 |
| **40% 미만** | 부족 | **반드시** 시각 요소 추가 |

### 7.3 빈 공간 충전 전략 (우선순위 순)

1. **관련 데이터 시각화** -- 수치를 차트/그래프/KPI 카드로 표현
2. **아이콘 그리드** -- 항목별 아이콘 + 라벨 조합
3. **이미지 플레이스홀더** -- 현장 사진, 시스템 스크린샷 영역
4. **보조 인포그래픽** -- 프로세스 플로우, 연결선, 화살표
5. **장식 도형** -- 대칭적 기하학 도형, 그라디언트 배경 블록
6. **인사이트 박스** -- 핵심 메시지를 강조하는 하단 요약 박스

> 장식 도형은 의미 있는 콘텐츠가 더 이상 없을 때만 사용합니다.
> 의미 없는 도형이라도 페이지의 시각적 균형을 위해 대칭(symmetric) 배치합니다.

### 7.4 빈 공간 충전 시각 요소 라이브러리

| 카테고리 | 요소 | 사용 조건 | 예시 |
|----------|------|----------|------|
| **데이터** | KPI 카드 | 관련 수치가 있을 때 | 숫자 + 단위 + 설명 카드 |
| **데이터** | 미니 차트 | 트렌드 데이터가 있을 때 | 소형 막대/라인 차트 |
| **정보** | 인사이트 박스 | 핵심 메시지 강조 시 | 하단 컬러 박스 + 텍스트 |
| **정보** | 인용 블록 | 임팩트 있는 한마디 | 큰따옴표 + 인용문 |
| **시각** | 아이콘 세트 | 항목에 이모지/아이콘 매칭 | 아이콘 + 라벨 그리드 |
| **시각** | 프로세스 화살표 | 흐름/연결 강조 시 | 단계 간 화살표 연결선 |
| **시각** | 이미지 플레이스홀더 | 관련 사진이 있을 때 | 둥근 모서리 사각형 |
| **장식** | 대칭 도형 | 콘텐츠 추가 불가 시 | 원형/사각형 대칭 배치 |
| **장식** | 그라디언트 블록 | 여백 채움 필요 시 | 색상 그라디언트 배경 |
| **장식** | 구분선/패턴 | 영역 구분 필요 시 | ACCENT 색상 얇은 선 |

### 7.5 시각적 풍성함 가이드 (Visual Richness)

> **단조로운 슬라이드의 원인**: 텍스트 카드만 나열하면 정보 전달은 되지만
> 제안서로서의 시각적 임팩트가 떨어집니다. 모든 슬라이드에 최소 하나 이상의
> **비텍스트 시각 요소**를 포함해야 합니다.

#### 7.5.1 필수 시각 요소 체크리스트

모든 본문 슬라이드는 아래 중 **최소 2가지 이상**을 포함해야 합니다:

| 카테고리 | 요소 | 시각적 효과 | pptxgenjs 구현 |
|----------|------|------------|---------------|
| **이미지 영역** | 사진 플레이스홀더 | 현장감, 신뢰감 | `addShape(RECTANGLE, {fill: 'D8DBE2'})` + 텍스트 |
| **아이콘 원형** | 원형 배경 + 숫자/아이콘 | 시선 집중, 단계 표시 | `addShape(OVAL, {fill: DOMINANT})` + 숫자 텍스트 |
| **액센트 바** | 좌측 세로 컬러 바 | 카드 구분, 계층감 | `addShape(RECTANGLE, {x, y, w:0.06, h})` |
| **구분선** | 수평 얇은 선 | 섹션 구분, 정돈감 | `addShape(RECTANGLE, {h:0.015, fill: ACCENT})` |
| **그라디언트 블록** | 다크/컬러 배경 영역 | 대비, 강조 | `addShape(RECTANGLE, {fill: DARK_BG})` |
| **인사이트 박스** | 하단 컬러 배경 + 텍스트 | 핵심 메시지 강조 | `addShape(RECTANGLE, {fill: SUBTLE})` |
| **태그/뱃지** | 작은 컬러 라벨 | 분류, 카테고리 표시 | `addShape(RECT, {h:0.28})` + 작은 텍스트 |
| **점수/지표 원형** | 원형 + 숫자 | 정량적 임팩트 | `addShape(OVAL)` + 큰 숫자 |
| **화살표/흐름선** | 방향 표시 | 프로세스, 인과 관계 | 텍스트 화살표 또는 `addShape(LINE)` |
| **장식 도형** | 투명 원형, 기하학 | 배경 질감, 깊이감 | `addShape(OVAL, {transparency: 85})` |
| **컬러 헤더 바** | 카드 상단 컬러 영역 | 카드 위계, 제목 강조 | `addShape(RECT, {fill: DOMINANT, h:0.5})` |
| **데이터 시각화** | 미니 차트, 도넛 | 정량 데이터 표현 | `addChart(PIE/BAR)` 또는 도형 조합 |

#### 7.5.4 장식 요소 배치 규칙

```
장식 도형은 콘텐츠를 방해하지 않는 위치에 배치합니다:

허용 위치:
- 표지/엔딩의 모서리 (투명 원형, 기하학적 패턴)
- 챕터 구분자의 우측 가장자리 (투명 바)
- 슬라이드 하단 양 모서리 (대칭 원형)
- 빈 영역의 배경 레이어 (높은 투명도)

금지 위치:
- 텍스트 위 (가독성 저하)
- 콘텐츠 카드 내부 (정보 혼란)
- 헤더 영역 (규격 위반)

투명도 규칙:
- 장식 원형: transparency 80~90
- 배경 바: transparency 85~95
- 콘텐츠 영역의 장식: transparency 90+
```

### 7.6 밀도별 대응 예시

**Case 1: 텍스트 3줄만 있는 슬라이드 (Fill-Rate ~20%)**
-> 좌측에 아이콘 그리드, 우측에 관련 이미지, 하단에 인사이트 박스 추가
-> 목표 Fill-Rate 70%

**Case 2: 카드 2개만 있는 슬라이드 (Fill-Rate ~35%)**
-> 카드를 3개로 확장하거나, 하단에 KPI 수치 카드 추가
-> 목표 Fill-Rate 65%

**Case 3: 테이블만 있는 슬라이드 (Fill-Rate ~50%)**
-> 적정 범위이나, 우측에 관련 프로젝트 카드 또는 인사이트 박스 추가 가능
-> 목표 Fill-Rate 65%

### 7.7 콘텐츠 밀도 밸런싱

> 제안서 전체의 시각적 밸런스를 위해 슬라이드 밀도를 교차 배치합니다.

| 밀도 등급 | 해당 슬라이드 | 특성 |
|----------|-------------|------|
| **LOW** (여백 많음) | 표지, 간지, 마무리 | 임팩트, 전환, 호흡 |
| **MID** (균형) | S05, S06, S07, S08, S19, S22, S40, S41 | 핵심 콘텐츠, 적정 정보량 |
| **HIGH** (정보 밀집) | S09, S14, S16, S29, S30, S31, S34 | 데이터, 테이블, 다수 항목 |

**밸런싱 규칙**:
- HIGH 슬라이드 2개 연속 금지 -> 사이에 MID 또는 LOW 삽입
- LOW 슬라이드는 챕터 전환부에 배치
- MID 슬라이드가 전체의 60% 이상 차지하도록 구성

## 9. STEP 4: 검수 및 수정 가이드

### 9.1 자동 검수 체크리스트

생성된 PPT를 열어 아래 항목을 확인합니다:

```markdown
## 검수 체크리스트

### 구조 검수
- [ ] 표지가 첫 페이지인가
- [ ] 목차가 표지 다음에 있는가 (선택)
- [ ] 각 챕터 시작에 간지가 있는가
- [ ] 마무리가 마지막 페이지인가
- [ ] 총 페이지 수가 계획과 일치하는가
- [ ] 마무리 페이지에 다짐 카피가 있는가 ("감사합니다"만 있지 않은가)

### 콘텐츠 검수
- [ ] 모든 헤더 메시지가 해당 슬라이드 내용을 정확히 요약하는가
- [ ] 페이지 번호(목차번호)가 올바르게 부여되었는가
- [ ] 목차 라벨이 해당 챕터명과 일치하는가
- [ ] 숫자/데이터가 원본 자료와 일치하는가
- [ ] 오탈자가 없는가

### 헤더 위치 검수 (최우선)
- [ ] 모든 본문 슬라이드의 핵심 메시지가 y=0.68in에 위치하는가
- [ ] 모든 본문 슬라이드의 부제목이 y=1.18in에 위치하는가
- [ ] 모든 본문 슬라이드의 콘텐츠가 y=1.77in 아래에서 시작하는가
- [ ] 헤더 영역의 좌우 위치가 x=0.47in, w=10.75in으로 통일되어 있는가
- [ ] 커스텀 슬라이드도 위 헤더 좌표를 동일하게 사용하는가
- [ ] 강조 페이지도 동일한 헤더 좌표를 사용하는가 (배경이 달라도 좌표 불변)

### 시각 밀도 검수
- [ ] 모든 본문 슬라이드의 빈 공간이 40% 이하인가
- [ ] 빈 공간이 40% 초과인 슬라이드에 시각 요소가 추가되었는가
- [ ] 장식 도형이 사용된 경우 대칭적으로 배치되어 있는가
- [ ] 콘텐츠 영역(w=10.63in, h=5.91in, y=1.77 시작)이 모든 페이지에서 동일한가

### 강조 페이지 비율 검수
- [ ] 전체 본문 대비 강조 페이지 비율이 5~12% 이내인가
- [ ] 강조 페이지가 연속 배치되어 있지 않은가

### 시각 검수
- [ ] 텍스트가 영역 밖으로 넘치지 않는가
- [ ] 이미지 플레이스홀더에 적절한 이미지가 들어갔는가
- [ ] 색상이 6:3:1 비율(DOMINANT 60%, SECONDARY 30%, ACCENT 10%)로 적용되었는가
- [ ] 색상이 제안 대상 브랜드 기반으로 결정되었는가
- [ ] HIGH 밀도 슬라이드가 2개 이상 연속되지 않았는가

### 논리 검수
- [ ] 챕터 간 논리적 흐름이 자연스러운가
- [ ] Before/After 내용이 실제로 대비되는가
- [ ] 프로세스 단계가 시간순으로 정렬되어 있는가
- [ ] KPI 수치가 일관성 있는가 (합계, 비율 등)
```

### 9.2 흔한 수정 사항

| 문제 | 원인 | 해결 |
|------|------|------|
| 텍스트 넘침 | 콘텐츠가 너무 많음 | 카드 설명을 1줄 줄이거나 슬라이드 분할 |
| 빈 영역 | 콘텐츠가 부족 | 보조 설명 추가 또는 더 적은 항목의 슬라이드로 교체 |

| 흐름 어색 | 슬라이드 순서 부적절 | 밀도 밸런싱 규칙 재적용 |
| 중복 표현 | 유사 슬라이드 다수 사용 | 하나로 통합하거나 다른 레이아웃으로 교체 |
| "감사합니다" 엔딩 | 마무리 카피 미작성 | 표지 카피를 다짐형으로 변주 |

## 10. 커스텀 슬라이드 제작 가이드

### 10.1 커스텀 슬라이드가 필요한 경우

1. **논리 구조가 기존 템플릿에 맞지 않을 때**
   - 예: 3개 문제를 종합하여 1개 공통 원인으로 수렴하는 구조
   - 예: 좌/우 비교가 아닌 3단 병렬 비교 + 하단 결론

2. **데이터 구조가 기존 항목 수와 다를 때**
   - 예: 5개 항목인데 3단 카드(S05)와 6칸 그리드(S19) 사이에 적합한 것이 없음

3. **복합 레이아웃이 필요할 때**
   - 예: 상단 프로세스 플로우 + 하단 비교 테이블 (단일 템플릿으로 불가)

### 10.2 커스텀 슬라이드 제작 규칙

```
절대 규칙:
1. 헤더 영역 (y=0.21 ~ 1.55in) -> 기존 템플릿과 100% 동일
2. 콘텐츠 영역 (y=1.77 ~ 7.68in) -> 자유롭게 설계
3. 좌우 마진 (x=0.53 ~ 11.16in) -> 기존 템플릿과 동일
4. 색상 체계 -> DOMINANT/SECONDARY/ACCENT 동일
5. 폰트 -> Pretendard 동일, 크기 체계 유지
6. Fill-Rate -> 60% 이상 유지
```

### 10.3 커스텀 프롬프트 작성법

```
### 슬라이드 #{순번} -- CUSTOM: {설명}
- 템플릿: CUSTOM (기존 S번호 없음 -- 신규 제작)
- 설계 의도: "{이 슬라이드가 논리 구조에서 하는 역할}"
- 페이지번호: {해당 목차번호}
- 목차라벨: "{챕터 제목}"

[헤더 영역 -- 좌표 고정]
- 핵심 메시지 (y=0.68in): "{핵심 메시지 1줄}"
- 부제목 (y=1.18in): "{보조 설명}"

[콘텐츠 영역 -- y=1.77in 이하, 레이아웃 설계]
- 레이아웃: {구체적 레이아웃 설명}
  예: "좌측 50% 문제 요약 카드 3장 세로 배치 +
      중앙 화살표 연결 +
      우측 50% 솔루션 카드 3장 대응 배치 +
      하단 인사이트 박스"
- {요소1}: {구체적 내용}
- {요소2}: {구체적 내용}

[시각 밀도]
- 예상 Fill-Rate: 70%
- 추가 시각 요소: {아이콘, 화살표, 장식 도형 등}
```

### 10.4 커스텀 레이아웃 패턴 라이브러리

| 패턴명 | 구조 | 적합한 콘텐츠 |
|--------|------|-------------|
| **수렴형** | 좌측 N개 카드 -> 중앙 화살표 -> 우측 1개 결론 | N개 문제 -> 1개 원인 |
| **대응형** | 좌측 N개 + 우측 N개 + 하단 결론 | 문제 <-> 해결책 1:1 매핑 |
| **계층형** | 상단 1개 핵심 -> 중단 3개 요소 -> 하단 상세 | 전략 -> 전술 -> 실행 |
| **매트릭스형** | 2x2 또는 3x2 그리드 + 축 라벨 | 2축 분류, 포지셔닝 맵 |
| **타임라인+카드형** | 상단 타임라인 -> 하단 각 단계별 카드 | 시간순 프로세스 + 상세 |
| **통계+인사이트형** | 좌측 대형 KPI + 우측 설명 카드 3~4개 | 핵심 수치 + 맥락 설명 |

## 11. 자료 부족 시 처리 프로세스 (A/B/C 등급)

> 사용자가 "만들어줘"라고 했을 때, 제공된 자료가 불충분하면
> PPT를 바로 만들지 않고 **필요한 정보를 먼저 확인**합니다.

### 11.1 자료 충분도 판단 기준

PPT 생성에 필요한 최소 정보를 아래 3등급으로 판단합니다:

| 등급 | 조건 | 행동 |
|------|------|------|
| **A: 즉시 생성** | 분석 자료(docx 등) + 제안 대상 명확 | 바로 STEP 1~3 실행 후 PPT 생성 |
| **B: 부분 확인** | 프로젝트명 + 방향은 있지만 상세 데이터 없음 | 핵심 질문 후 PPT 생성 |
| **C: 기획부터** | 프로젝트명만 있음 | 기획 질문 -> 구조 합의 -> PPT 생성 |

### 11.2 등급별 질문 프로세스

#### 등급 B: 부분 확인 (3~5개 질문)

사용자가 프로젝트 방향은 알려줬지만 상세가 없을 때:

```
필수 확인 항목:
1. [제안 대상] 제안서를 받는 업체/담당자는 누구인가요?
   -> 색상 결정 + 톤앤매너에 영향
2. [핵심 문제] 해결하려는 문제/과제가 구체적으로 무엇인가요?
   -> 챕터 구조 + 과제 진단 페이지에 영향
3. [솔루션] 우리가 제안하는 해결책의 핵심은 무엇인가요?
   -> 솔루션 챕터 구성에 영향
4. [실적] 관련 레퍼런스나 수치 데이터가 있나요?
   -> KPI/실적 페이지에 영향
5. [범위] 예상 페이지 수나 분량 선호가 있나요?
   -> 전체 구조 + 4분류 배분에 영향
```

#### 등급 C: 기획부터 (5~8개 질문)

사용자가 프로젝트명만 알려줬을 때:

```
기획 확인 항목:
1. [배경] 이 제안을 하게 된 배경/계기는 무엇인가요?
2. [제안 대상] 누구에게 보내는 제안서인가요?
3. [목표] 이 제안서로 달성하려는 목표는 무엇인가요?
4. [현재 문제] 현재 어떤 문제/과제가 있나요?
5. [우리 솔루션] 우리가 제안하는 해결 방향은 무엇인가요?
6. [차별점] 경쟁사 대비 우리의 강점은 무엇인가요?
7. [실적] 관련 프로젝트 경험이나 수치가 있나요?
8. [기타] 반드시 포함해야 할 내용이나 피해야 할 내용이 있나요?
```

### 11.3 질문 후 처리 흐름

```
사용자 입력: "펜타포트 셔틀 제안서 만들어줘"
     |
     v

     |

     |
```

### 12.1 코드 생성 기본 구조

```javascript
const pptxgen = require("pptxgenjs");
const pres = new pptxgen();

// === 1. 슬라이드 규격 ===
pres.defineLayout({ name: "A4L", width: 11.69, height: 8.27 });
pres.layout = "A4L";
pres.author = "{회사명}";
pres.title = "{제안서 제목}";

// === 2. 색상 변수 (6:3:1 시스템 -- 제안 대상에 따라 변경) ===
const DOM = '{DOMINANT}';    // 60% -- 간지 배경, 넓은 면적
const SEC = '{SECONDARY}';   // 30% -- 카드, 태그
const ACC = '{ACCENT}';      // 10% -- 핵심 수치, CTA
const DK  = '{DARK_BG}';     // 표지/마무리 배경
const LT  = '{LIGHT_BG}';    // 카드 배경
const CD  = '{CARD_BG}';     // 테이블 교차행
const SB  = '{SUBTLE}';      // 인사이트 박스
const W   = 'FFFFFF';
const TD  = '1A1A2E';        // 본문 텍스트
const TG  = '6B7280';        // 보조 텍스트
const TF  = '{TEXT_FAINT}';  // 표지/간지 보조 텍스트

// ━━━ 폰트 (가중치별 — 마스터 템플릿 실측 기반) ━━━
const FN  = 'Pretendard';           // 기본 (Regular — 본문 텍스트용)
const FN_XB = 'Pretendard ExtraBold'; // ExtraBold (fontFace 지정): 표지 메인카피, 간지 타이틀, 마무리 다짐카피, 본문 헤더 메시지(16pt), 카드 제목

const FN_TN = 'Pretendard Thin';      // 표지 서브카피, 간지 보조, 마무리 저작권
// ★ 참고: Pretendard 폰트가 가중치별로 설치되어 있지 않으면 FN으로 폴백

// === 3. 공통 헬퍼 함수 ===
// (아래 섹션 12.2~12.5에서 정의)

// === 4. 슬라이드 생성 (4분류별) ===
// (아래 섹션 12.6에서 정의)
```

```javascript
// 모든 본문 페이지(기본+강조)에서 반드시 사용
function hdr(sl, pg, toc, msg, sub) {
  // 페이지번호 + 목차라벨
  sl.addText([
    { text: pg, options: { fontSize: 10, color: DOM, bold: true, fontFace: FN } },
    { text: '  ' + toc, options: { fontSize: 9, color: TG, fontFace: FN } }
  ], { x: 0.47, y: 0.21, w: 5.0, h: 0.25, margin: 0 });
  // 핵심 메시지 (ExtraBold — fontFace 지정, bold:true 금지 AP-15)
  sl.addText(msg, {
    x: 0.47, y: 0.68, w: 10.75, h: 0.47,
    fontSize: 16, fontFace: FN_XB, color: TD, margin: 0
  });
  // 부제목
  if (sub) sl.addText(sub, {
    x: 0.47, y: 1.18, w: 10.75, h: 0.29,
    fontSize: 10, fontFace: FN, color: TG, margin: 0
  });
  // 구분선 (DOM — brand primary)
  sl.addShape(pres.shapes.RECTANGLE, {
    x: 0.47, y: 1.55, w: 10.75, h: 0.015, fill: { color: DOM }
  });
  // 하단 페이지 번호
  sl.addText(pg, {
    x: 10.50, y: 7.68, w: 0.70, h: 0.30,
    fontSize: 10, fontFace: FN, color: TG, align: 'right', margin: 0
  });
}
```

### 12.3 헬퍼 함수: 간지 (DIVIDER)

> **마스터 S02 실측 기반.** 좌측마진 0.57", 라벨→타이틀→보조 3단 구성.
> 핵심 차별점: 타이틀은 **40pt ExtraBold**로 강한 존재감.

```javascript
function divider(label, num, copyLine, subLine, bgData) {
  const sl = pres.addSlide();
  // ── 배경: AI 그래디언트 있으면 사용, 없으면 DOM 단색 ──
  if (bgData) {
    sl.background = { data: bgData };
  } else {
    sl.background = { color: DOM };
  }
  // ── 챕터 라벨 (예: "CHAPTER 01") ──
  sl.addText(label + ' ' + num, {
    x: 0.57, y: 2.33, w: 8.77, h: 0.32,
    fontSize: 13, fontFace: FN_MD, color: SEC, charSpacing: 3, margin: 0
  });
  // ── 챕터 타이틀 (메시지성 카피 — 2줄까지 허용) ──
  sl.addText(copyLine, {
    x: 0.57, y: 3.03, w: 8.0, h: 1.45,
    fontSize: 40, fontFace: FN_XB, color: W,
    lineSpacingMultiple: 1.2, margin: 0
  });
  // ── 보조 설명 (1~2줄) ──
  sl.addText(subLine, {
    x: 0.57, y: 4.47, w: 8.77, h: 0.54,
    fontSize: 13, fontFace: FN_TN, color: TF, lineSpacingMultiple: 1.4, margin: 0
  });
  // ── 장식: 우측 투명 바 ──
  sl.addShape(pres.shapes.RECTANGLE, {
    x: 10.5, y: 0, w: 1.19, h: 8.27,
    fill: { color: SEC, transparency: 90 }
  });
  // ── 장식: 하단 라인 ──
  sl.addShape(pres.shapes.RECTANGLE, {
    x: 0, y: 7.87, w: 11.69, h: 0.04,
    fill: { color: SEC, transparency: 60 }
  });
  return sl;
}
```

### 12.5 4분류별 코드 패턴

#### 표지 (COVER)

> **마스터 S01 실측 기반.** 배경 #0D0D2B + DOM 오버레이(30%투명).
> 좌측마진 0.57". 3단 텍스트: 라벨(18pt)→메인카피(40~54pt)→서브(14pt).
> AI 모듈의 배경 이미지(bgData)가 있으면 그래디언트+장식 배경으로 대체.

```javascript
function createCover(impactCopy, subLabel, subCopy, info, bgData) {
  const sl = pres.addSlide();

  // ── 배경: AI 생성 배경 또는 마스터 스타일 재현 ──
  if (bgData) {
    sl.background = { data: bgData };  // ai_brand_analyzer의 gradients.cover
  } else {
    sl.background = { color: DK };
    // 마스터 S01 오버레이: DOM 30% 투명 전면 사각형
    sl.addShape(pres.shapes.RECTANGLE, {
      x: 0, y: 0, w: 11.69, h: 8.27,
      fill: { color: DOM, transparency: 70 }
    });
  }

  // ── 상단 라벨 (예: "OOO 제안서") ──
  sl.addText(subLabel || '', {
    x: 0.57, y: 2.43, w: 8.77, h: 0.40,
    fontSize: 18, fontFace: FN_MD, color: SEC, margin: 0
  });

  // ── 메인 임팩트 카피 (ExtraBold, 최대 2줄) ──
  // ★ 1줄이면 54pt, 2줄이면 40pt로 자동 조절 권장
  const lines = (impactCopy || '').split('\n').length;
  sl.addText(impactCopy, {
    x: 0.57, y: 2.91, w: 9.50, h: 1.92,
    fontSize: lines > 1 ? 40 : 54, fontFace: FN_XB, color: W,
    lineSpacingMultiple: 1.2, margin: 0
  });

  // ── 서브 카피 (1줄 보조 설명) ──
  sl.addText(subCopy || '', {
    x: 0.57, y: 4.83, w: 8.77, h: 0.34,
    fontSize: 14, fontFace: FN_TN, color: TF, margin: 0
  });

  // ── 하단 제안 정보 ──
  sl.addShape(pres.shapes.RECTANGLE, {
    x: 0, y: 7.4, w: 11.69, h: 0.015, fill: { color: SEC, transparency: 50 }
  });
  sl.addText(info || '', {
    x: 0.57, y: 7.50, w: 10, h: 0.30,
    fontSize: 9, fontFace: FN_TN, color: TF, margin: 0
  });

  return sl;
}

// 사용 예시:
createCover(
  '제네시스를 경험하는 모든 이동,\n그라운드케이가 설계합니다',  // 메인 카피 (2줄→40pt)
```

#### 마무리 (ENDING)

> **마스터 S43 실측 기반.** 배경 DOM(#2E3092). **좌정렬** x=1.47" (표지/간지보다 안쪽).
> 다짐카피 36pt ExtraBold + 구분선 2" + 부연 14pt Medium.
> ★ 절대 "감사합니다"만 넣지 않는다 — 반드시 다짐/서약형 카피.

```javascript
function createEnding(pledgeCopy, closingMsg, copyright, bgData) {

  if (bgData) {
    sl.background = { data: bgData };  // ai_brand_analyzer의 gradients.ending

  sl.addText(closingMsg || '', {
```

### 12.6 코드 생성 자동화 흐름

```
[STEP 3에서 결정된 슬라이드 리스트]
     |
     v
각 슬라이드의 '분류'를 확인:
  - 표지 -> createCover() 호출
  - 간지 -> divider() 호출
  - 본문-기본 -> createStandard() + 콘텐츠 빌더
  - 본문-강조 -> createHighlight() + 콘텐츠 빌더
  - 마무리 -> createEnding() 호출
     |
     v
각 콘텐츠 빌더에서:
  1. 기존 템플릿(S04~S41) 레이아웃 참조 -> 해당 패턴의 좌표/구조 적용
  2. CUSTOM인 경우 -> 섹션 10의 커스텀 레이아웃 패턴 참조
  3. 시각 요소 2가지+ 포함 확인
  4. Fill-Rate 60%+(기본) 또는 75%+(강조) 확인
     |
     v
pres.writeFile() -> .pptx 출력
```

| 팀 소개에 S05(3단카드) 사용 | 유사 구조이지만 전용 있음 | S22(팀 전용) |
| SWOT에 S09(테이블) 사용 | 전용 슬라이드 존재 | S26(SWOT 전용) |
| KPI에 S29(정보그리드) 사용 | 데이터 특성 불일치 | S12/S27/S38(KPI 전용) |

### 13.2 콘텐츠 작성 실수

| 실수 | 문제 | 해결 |
|------|------|------|
| 헤더에 30자 이상 | 텍스트 넘침 | 15~25자로 압축 |
| 카드 설명이 4줄 이상 | 가독성 저하 | 2~3줄로 요약, 핵심만 |
| KPI 숫자에 단위 생략 | 의미 불명확 | 반드시 단위 포함 (건, 명, %) |
| Before 3개 / After 2개 | 비대칭 | 양쪽 동일 항목 수 유지 |
| 프로세스 단계 설명이 중복 | 각 단계 차별화 부족 | 단계별 고유 활동 명시 |

### 13.3 구조 배치 실수

| 실수 | 문제 | 해결 |
|------|------|------|
| 테이블 3개 연속 | 시각적 피로 | 사이에 카드형/이미지형 삽입 |
| 챕터 구분 없이 15장 | 읽기 흐름 단절 | 4~6장마다 간지 삽입 |
| 표지 직후 본문 | 전환 어색 | 반드시 목차 또는 간지 삽입 |
| 서브엔딩+엔딩 연속 | 마무리 중복 | 서브엔딩은 최종 챕터 끝에만 |
| 마무리에 "감사합니다"만 | 임팩트 없음 | 표지 카피를 다짐형으로 변주 |
| 강조 페이지 20%+ | 강조 과다 | 5~10%로 줄이고 기본으로 전환 |

### 13.4 좌표/규격 실수

| 실수 | 문제 | 해결 |
|------|------|------|
| 콘텐츠를 y=1.67부터 배치 | 헤더와 겹침 위험 | y=1.77부터 시작 |
| 헤더 좌표가 페이지마다 다름 | 일관성 파괴 | 섹션 6 좌표표 참조, hdr() 함수 사용 |
| 강조 페이지에서 헤더 생략 | 규격 위반 | 배경이 달라져도 헤더 좌표 유지 |
| 콘텐츠가 y=7.68 아래로 넘김 | 하단 페이지번호와 충돌 | y=7.68 이내로 제한 |
